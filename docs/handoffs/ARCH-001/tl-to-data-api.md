# ARCH-001 TL → DATA/API 인수인계

## 전달 목적

DATA-001과 API-001이 첫 번째 수직 MVP의 저장 구조와 API 계약을 설계할 때 사용할 아키텍처 입력을 전달한다.

제품·도메인·UX에서 확정된 것과 아직 후속 설계가 정해야 할 것을 분리해, DB 설계와 API 계약이 승인되지 않은 정책을 새로 만들지 않도록 한다.

## 승인된 입력

- 상품 목록은 비회원과 로그인 회원 모두 조회할 수 있는 공개 기능이다.
- 상품 상세, SKU와 구독 가능 여부 확인은 공개 기능이다.
- 구독 생성, 내 구독 목록, 내 구독 상세는 로그인한 회원만 사용할 수 있다.
- 로그인 성공 후 안전한 서비스 내부 GET 경로가 있으면 해당 경로로 복귀하고, 없으면 상품 목록으로 이동한다.
- 외부 URL과 검증되지 않은 임의 경로로 이동하지 않는다.
- 입력 폼 상태 자동 복원과 이전 POST 요청 자동 재실행은 첫 번째 MVP에서 제외한다.
- 구독 하나는 SKU 하나만 대상으로 한다.
- 수량은 1~10이다.
- 배송 주기는 2주, 4주, 8주다.
- 구독 생성 성공 후 생성된 구독 상세 화면으로 이동한다.
- 별도의 영구적인 구독 생성 성공 페이지는 만들지 않는다.
- 생성 전 정확한 다음 주문 예정일 날짜를 표시하지 않는다.
- 생성 후, 내 구독 목록, 내 구독 상세에서는 서버가 확정한 다음 주문 예정일을 표시한다.
- 다음 주문 예정일은 `Asia/Seoul` 기준 날짜 단위로 계산한다.
- 휴일·주말·영업일 보정은 적용하지 않는다.
- 다음 배송 예정일은 표시하지 않는다.

## 아키텍처 결정 요약

- 시스템 경계는 `사용자 → 프론트엔드 → 백엔드 API → 도메인/애플리케이션 서비스 → 영속성`으로 나눈다.
- 공개 조회와 보호 기능을 분리한다.
- 인증은 보호 기능 접근 여부를 결정하고, 인가는 본인 구독 조회 여부를 결정한다.
- 도메인은 구독 생성 불변 조건과 다음 주문 예정일 계산 책임을 가진다.
- 애플리케이션은 인증 회원 식별, SKU 조회, 구독 생성 흐름, 소유자 검증, 트랜잭션 조정을 담당한다.
- 프론트엔드는 생성 전 다음 주문 예정일을 계산하지 않고 서버 확정 값을 표시한다.
- API-001은 HTTP 상태, 오류 코드, 응답 JSON과 날짜 표현을 확정한다.
- DATA-001은 저장 구조, 관계, 제약과 조회 인덱스 후보를 확정한다.

## DATA-001 입력

- Product, SKU, Member, Subscription의 개념 저장 구조를 검토한다.
- Product는 상품 공통 정보와 SKU 목록 또는 SKU별 표시 정보를 제공해야 한다.
- SKU는 용량 또는 구성, 표시 가격, 구독 가능 여부를 제공해야 한다.
- Subscription은 회원, SKU 하나, 수량, 배송 주기, 구독 생성일, 다음 주문 예정일을 보존해야 한다.
- Subscription은 회원 소유자를 식별할 수 있어야 한다.
- Subscription이 참조할 회원과 SKU 관계를 정한다.
- 수량 1~10 제약을 저장 구조 또는 애플리케이션/도메인 검증과 어떻게 함께 보장할지 정한다.
- 배송 주기 2주, 4주, 8주 저장 방식을 정한다.
- 구독 생성일과 다음 주문 예정일 저장 여부와 타입을 정한다.
- SKU 구독 가능 여부 저장 방식을 정한다.
- 내 구독 목록과 상세 조회를 위한 인덱스 후보를 검토한다.
- Product와 SKU를 같은 저장 단위로 둘지 별도 저장 단위로 둘지 검토한다.
- Subscription이 SKU 전체를 참조할지 SKU 식별자와 필요한 스냅샷 의미를 보유할지 검토한다.

DATA-001에서 아직 하지 말아야 할 것:

- API 응답 JSON 확정
- HTTP 상태와 오류 코드 확정
- 프론트엔드 라우트 확정
- 인증 구현 방식 확정

## API-001 입력

- 상품 목록 조회 계약을 정한다.
- 상품 상세·SKU 조회 계약을 정한다.
- 구독 생성 요청·응답 계약을 정한다.
- 내 구독 목록 조회 계약을 정한다.
- 내 구독 상세 조회 계약을 정한다.
- 상품 목록, 상품 상세, SKU와 구독 가능 여부 확인은 공개 조회로 표현한다.
- 구독 생성, 내 구독 목록, 내 구독 상세는 인증 필요 기능으로 표현한다.
- 내 구독 목록과 상세에는 본인 구독만 포함되도록 인가 실패 표현을 정한다.
- 구독 생성 성공 응답에는 후속 상세 이동에 필요한 구독 식별자가 포함되어야 한다.
- 생성 성공 후 서버가 확정한 다음 주문 예정일을 제공해야 한다.
- 날짜 표현 형식을 정한다.
- 오류 코드와 HTTP 상태를 정한다.
- 존재하지 않는 구독과 다른 회원 소유 구독이 사용자에게 같은 “조회할 수 없는 구독”으로 표현되도록 계약을 정한다.

API-001에서 다룰 오류 범주:

- 비로그인 보호 기능 접근
- 존재하지 않는 상품
- 존재하지 않는 SKU
- 구독 불가능 SKU
- 수량 범위 위반
- 허용되지 않은 배송 주기
- 필수 입력 누락
- 존재하지 않는 구독
- 다른 회원 소유 구독 접근
- 일시적 조회 실패

## 인증 설계 입력

- 보호 기능 접근으로 로그인 화면에 이동한 경우 로그인 성공 후 안전한 서비스 내부 GET 경로로 복귀한다.
- 유효한 내부 복귀 경로가 없으면 상품 목록으로 이동한다.
- 외부 URL과 검증되지 않은 임의 경로로 이동하지 않는다.
- 입력 폼 상태를 자동 복원하지 않는다.
- 이전 POST 요청을 자동으로 다시 실행하지 않는다.
- 로그인 복귀는 인증 설계의 책임이다.
- Open Redirect 방지는 반드시 후속 인증 설계에서 검증한다.

인증 설계가 정해야 할 것:

- 복귀 경로 저장 방식
- 복귀 경로 저장 위치와 수명
- 안전한 내부 GET 경로 판별 방식
- 허용 경로 목록 또는 검증 방식
- 인증 실패와 인가 실패의 구현 방식
- Spring Security 또는 동등한 인증·인가 구현 방식

## FE/QA 참고 입력

- FE는 상품 목록과 상품 상세를 공개 탐색 화면으로 구현한다.
- FE는 보호 기능 접근 시 로그인 이동 흐름을 구현한다.
- FE는 로그인 후 안전한 내부 GET 복귀를 반영하되 입력 폼 상태 자동 복원과 이전 POST 자동 재실행을 하지 않는다.
- FE는 구독 생성 전 정확한 다음 주문 예정일 날짜를 표시하지 않는다.
- FE는 구독 생성 성공 후 생성된 구독 상세 화면으로 이동한다.
- FE는 서버 확정 다음 주문 예정일을 `YYYY. M. D.` 형식으로 표시한다.
- QA는 공개 상품 목록·상세 접근, 보호 기능 인증 요구, 다른 회원 구독 접근 차단, 구독 생성 정상·실패 조건, 생성 후 상세 이동, 생성 전 예정일 미표시, 날짜 기준과 보정 없음 기준을 검증한다.

## Deferred Technical Decision

- 실제 테이블명, 컬럼명, 외래키, 인덱스
- 배송 주기와 구독 가능 여부의 DB 저장 표현
- 날짜 DB 타입
- API 요청·응답 JSON
- HTTP 상태와 오류 코드
- 인증 실패와 인가 실패의 API 표현
- 날짜 API 표현
- 상품 목록, 상품 상세, 구독 상세, 로그인 화면의 구체 URI와 라우트 문자열
- 로그인 복귀 경로 저장 위치와 수명
- 안전한 서비스 내부 GET 경로 판별 방식
- Open Redirect 방지 구현
- Spring Security 또는 동등한 인증·인가 구현 방식
- 프론트엔드 라우팅 구현 방식
- 날짜 공급 추상화 사용 여부

## 중단 조건

- 제품 정책 재결정이 필요한 경우
- DB 테이블·컬럼을 확정하지 않고는 진행할 수 없는 경우
- API 요청·응답 JSON이나 HTTP 상태를 확정하지 않고는 진행할 수 없는 경우
- 인증 구현 방식을 확정하지 않고는 진행할 수 없는 경우
- 코드 변경이나 신규 의존성이 필요한 경우
- PS-002, DOMAIN-001, PS-003, UX-002 사이에 직접 충돌이 발견되는 경우
- Secret 또는 민감정보 노출이 의심되는 경우
