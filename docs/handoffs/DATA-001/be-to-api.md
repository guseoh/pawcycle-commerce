# DATA-001 Backend → API 인수인계

## 전달 목적

API-001이 첫 번째 수직 MVP의 공개 상품 조회, 구독 생성, 내 구독 목록·상세 조회 계약을 설계할 때 사용할 데이터 설계 입력을 전달한다.

DATA-001에서 제안한 데이터 구조와 API-001에서 확정해야 할 요청·응답, 오류 표현, 날짜 표현을 분리한다.

## 승인된 입력

- 상품 목록과 상품 상세는 비회원과 로그인 회원 모두 조회할 수 있는 공개 기능이다.
- SKU와 구독 가능 여부 확인은 공개 탐색 범위다.
- 구독 생성, 내 구독 목록, 내 구독 상세는 로그인한 회원만 사용할 수 있다.
- 구독 하나는 SKU 하나만 대상으로 한다.
- SKU는 구독 가능 여부를 가진다.
- 수량은 1~10이다.
- 배송 주기는 2주, 4주, 8주다.
- 구독 생성 성공 후 생성된 구독 상세 화면으로 이동한다.
- 생성 성공 후 구독 상세에서 구독 ID와 서버가 확정한 다음 주문 예정일을 표시한다.
- 구독 생성 전에는 정확한 다음 주문 예정일 날짜를 표시하지 않는다.
- 내 구독 목록과 상세에서도 서버가 확정한 다음 주문 예정일을 표시한다.
- 날짜 계산과 표시는 `Asia/Seoul` 기준 날짜 단위다.
- 휴일, 주말, 영업일 보정은 적용하지 않는다.
- 다음 배송 예정일은 표시하지 않는다.
- 구독 상태, 결제, 재고, 배송, 구독 변경 기능은 첫 MVP에서 제외한다.

## 데이터 설계 요약

| 개념 | 테이블 후보 | API 관점의 의미 |
| --- | --- | --- |
| Member | `members` | 인증된 회원 식별자와 구독 소유자 검증 기준 |
| Product | `products` | 상품 목록·상세의 공통 표시 정보 |
| SKU | `skus` | 실제 구독 선택 단위, 가격, 용량 또는 구성, 구독 가능 여부 |
| Subscription | `subscriptions` | 회원이 SKU 하나로 만든 구독, 수량, 주기, 생성일, 다음 주문 예정일 |

주요 관계:

- Product 1:N SKU
- Member 1:N Subscription
- SKU 1:N Subscription

주요 제약 후보:

- `subscriptions.quantity`: 1 이상 10 이하
- `subscriptions.delivery_cycle_weeks`: 2, 4, 8 중 하나
- `subscriptions.member_id`: nullable false, FK 후보
- `subscriptions.sku_id`: nullable false, FK 후보
- `skus.subscribable`: nullable false
- `subscriptions.next_order_date`: 서버 계산 확정값, nullable false 후보

## API-001 입력

API-001은 다음 계약을 확정해야 한다.

- 상품 목록 응답에 SKU 구독 가능 요약을 어느 수준까지 포함할지
- 상품 목록 응답 후보에서 상품 ID, 상품명, 대상 반려동물, 짧은 설명, 대표 이미지, SKU 구독 가능 요약, SKU별 표시 가격을 어떻게 표현할지
- 상품 상세 응답의 Product/SKU 필드 구조
- 상품 상세 응답 후보에서 상품 설명, SKU 목록, SKU별 가격, 구독 가능 여부, 선택 가능한 배송 주기 표현을 어떻게 제공할지
- SKU의 `subscribable` 표현명과 구독 불가능 이유 노출 여부
- 구독 생성 요청 필드명
- 구독 생성 성공 응답에 포함할 구독 ID와 다음 주문 예정일 표현
- 생성 성공 후 상세 이동에 필요한 식별자 제공 방식
- 내 구독 목록 응답 구조
- 내 구독 상세 응답 구조
- 날짜 표현 형식
- 인증 실패와 인가 실패 HTTP 상태
- 도메인 오류 코드와 메시지 구조
- 존재하지 않는 구독과 다른 회원 소유 구독을 같은 사용자 표현으로 보장하는 방식

위 항목은 API-001에서 검토할 응답 후보와 필수 표시 데이터다. DATA-001은 API 요청·응답 JSON 구조, 필드명, 중첩 구조, HTTP 상태, 오류 코드와 메시지 구조를 확정하지 않는다.

API-001에서 다룰 오류 범주:

| 오류 조건 | 데이터 근거 | API-001 결정 |
| --- | --- | --- |
| 존재하지 않는 상품 | `products`에 id 없음 | HTTP 상태, 오류 코드, 응답 JSON |
| 존재하지 않는 SKU | `skus`에 id 없음 | HTTP 상태, 오류 코드, 응답 JSON |
| 구독 불가능 SKU | `skus.subscribable=false` | HTTP 상태, 오류 코드, 사용자 메시지 |
| 필수 입력 누락 | `sku_id`, `quantity`, `delivery_cycle_weeks` 요청 값 없음 | 필드 오류 표현 |
| 수량 범위 위반 | `quantity` 1~10 위반 | 필드 오류 표현 |
| 허용되지 않은 배송 주기 | `delivery_cycle_weeks`가 2/4/8 아님 | 필드 오류 표현 |
| 존재하지 않는 구독 | `subscriptions`에 id 없음 | 조회할 수 없는 구독 표현 |
| 다른 회원 소유 구독 접근 | `subscriptions.member_id` 불일치 | 조회할 수 없는 구독 표현과 정보 노출 방지 |

## Backend 구현 입력

Backend 구현은 DATA-001을 바탕으로 다음을 확정해야 한다.

- JPA Entity 클래스 구조
- 연관관계 방향
- fetch 전략
- cascade 사용 여부
- 도메인 값 객체 사용 여부
- enum 저장 방식
- `subscribable` 저장 타입 최종 선택
- 트랜잭션 경계
- Repository 쿼리 방식
- Flyway 마이그레이션 DDL
- MySQL CHECK 제약 실제 적용 여부
- 날짜 DB 타입 최종 선택
- `Clock` 또는 동등한 날짜 공급 추상화 사용 여부

Backend 구현 시 유지해야 할 데이터 규칙:

- 구독 하나는 SKU 하나만 참조한다.
- 구독 생성 시 SKU가 존재하고 구독 가능해야 한다.
- 수량은 1~10이어야 한다.
- 배송 주기는 2주, 4주, 8주 중 하나여야 한다.
- 생성일과 다음 주문 예정일은 `Asia/Seoul` 날짜 단위 기준이어야 한다.
- 다음 주문 예정일은 서버가 계산한 확정값이어야 한다.
- 내 구독 목록과 상세는 `member_id` 소유자 조건을 적용해야 한다.

## QA 참고 입력

QA는 다음 데이터 기준을 검증 항목으로 사용할 수 있다.

- 상품 목록에서 공개 상품과 구독 가능 SKU 요약을 확인할 수 있다.
- 상품 상세에서 SKU별 가격, 용량 또는 구성, 구독 가능 여부를 확인할 수 있다.
- 구독 가능한 SKU, 수량 1~10, 배송 주기 2/4/8이면 구독 데이터가 생성될 수 있다.
- 구독 불가능 SKU로는 구독 데이터가 생성되면 안 된다.
- 존재하지 않는 SKU로는 구독 데이터가 생성되면 안 된다.
- 수량 0, 11 등 범위를 벗어난 값으로 구독 데이터가 생성되면 안 된다.
- 허용되지 않은 배송 주기로 구독 데이터가 생성되면 안 된다.
- 생성된 구독은 회원과 SKU 하나를 참조해야 한다.
- 생성된 구독은 구독 생성일과 다음 주문 예정일을 가져야 한다.
- 내 구독 목록은 본인 구독만 조회할 수 있어야 한다.
- 다른 회원의 구독 상세는 조회할 수 없어야 한다.
- 다음 주문 예정일은 휴일·주말·영업일 보정 없이 서버 계산 확정값이어야 한다.
- 다음 배송 예정일, 구독 상태, 결제, 재고, 배송 데이터는 첫 MVP 구독 응답에 포함되지 않아야 한다.

## Deferred Technical Decision

- 실제 SQL DDL
- Flyway 마이그레이션 파일명과 내용
- JPA Entity 구조
- 테이블명과 컬럼명의 최종 확정 여부
- enum 저장 방식
- 날짜 DB 타입 최종 선택
- `created_at`/`updated_at` 공통 감사 필드 구현 방식
- 인증 테이블과 권한 모델
- API 요청·응답 JSON
- HTTP 상태와 오류 코드
- URL과 라우트 문자열
- 성능 측정 전 최종 인덱스 확정
- Product와 SKU Aggregate 경계
- Subscription과 SKU 참조 방식

## 중단 조건

- API 요청·응답 JSON을 확정하려면 추가 제품 결정이 필요한 경우
- HTTP 상태와 오류 코드 결정이 승인된 제품 규칙과 충돌하는 경우
- 결제, 재고, 배송, 구독 상태 모델을 추가해야만 API 계약을 설계할 수 있는 경우
- 인증 구현 방식 또는 Spring Security 구성을 확정해야만 DATA-001 범위를 설명할 수 있는 경우
- 신규 의존성 또는 구현 코드 변경이 필요한 경우
- Secret 또는 민감정보 노출이 의심되는 경우
