# 위험 기반 Lean Harness

## 상태와 권위

- 상태: Approved
- 승인 작업: `HARNESS-LEAN-001`
- 적용 시점: 이 작업이 `main`에 병합된 뒤 시작하는 작업

이 문서는 작업 등급, 승격 기준, 등급별 산출물·QA·검증, delta-only Codex 명세 규칙의 상세 권위 원본이다. 루트·경로별 `AGENTS.md`, 역할 문서와 Skill, PR·보고서·인수인계 template, 협업 자동화 런북은 이 문서를 요약하거나 참조한다. 요약과 이 문서가 충돌하면 이 문서를 따른다.

기존 작업 기록과 과거 보고서·인수인계는 소급 수정하거나 삭제하지 않는다. 등급 필드가 없는 기존 산출물은 validator가 하위 호환 일반 규칙으로 검사한다.

## 작업 등급

| 등급 | 판정 기준 | 기본 처리 |
| --- | --- | --- |
| 경량 | 외부 계약이 불변인 한 역할의 작은 변경 | 한 PR, 보고서·인수인계 기본 생략, 변경 표면의 집중 검증 |
| 일반 | 하나의 사용자 목적을 완성하는 기능군 | 하나의 기능 ID, 필요한 역할 PR과 실질 보고서, 실제 소비자가 있을 때만 인수인계, 필요한 마지막 QA |
| 고위험 | 인증·결제·migration·배포·보안·성능·백업·복구·외부 서비스·공통 CI/CD | 명시적 승인, 필수 보고서, 적용 전후·독립 검증, 복구·롤백 증거 |

등급이 불명확하면 일반으로 시작한다. 작업 중 외부 계약, 데이터, 보안, 운영 또는 복구 위험을 발견하면 고위험으로 승격하고 승인·검증 계획을 보완한 뒤 계속한다. 경량 작업이 여러 역할, 사용자 동작 또는 외부 계약을 건드리면 일반 이상으로 승격한다. 등급 하향은 위험이 없다는 근거와 사용자 또는 Tech Lead 확인 없이 수행하지 않는다.

## 산출물과 인수인계

### 경량

- PR 본문에 작업 ID, 작업 등급, 목적, 변경 범위, 검증 결과와 남은 위험을 기록한다.
- `docs/reports/<작업 ID>/`와 `docs/handoffs/<작업 ID>/`는 기본 생략한다.
- 후속 역할이 실제로 사용할 계약·결정·실행 조건이 생기면 인수인계를 작성하고 일반으로 승격할지 검토한다.

### 일반

- 역할 PR마다 복제 문구가 아닌 실질 변경·검증·위험을 담은 작업 보고서를 작성한다.
- 인수인계는 다음 역할이 결과를 실제 입력으로 사용할 때만 작성한다. 소비자가 없으면 보고서의 `인수인계 생략`에 이유를 기록한다.
- 같은 기능군은 하나의 기능 ID를 사용한다. 역할별 별도 ID를 만들지 않는다.

### 고위험

- 작업 보고서는 필수다.
- 보고서에는 `작업 등급`, `명시적 승인 근거`, `적용 전 검증`, `적용 후 검증`, `독립 검증`, `복구·롤백 증거`를 내용과 함께 기록한다.
- 다음 역할 또는 운영자가 변경 결과를 사용하면 인수인계를 작성한다. 실제 소비자가 없으면 보고서에 생략 이유를 명시한다.
- 복구·롤백은 실행 가능한 경로와 검증 결과를 기록한다. 복구 절차 자체가 데이터 손실이나 history rewrite를 요구하면 중단하고 별도 승인을 받는다.

## QA와 독립 검증

- 경량은 변경 표면의 자동·수동 집중 검증을 수행하며 별도 QA 문서는 기본 요구하지 않는다.
- 일반은 중간 역할마다 형식적 QA를 반복하지 않는다. 사용자 목적이 완성되는 마지막 의미 있는 변경에서 QA 필요 여부를 판단하고, 필요하면 기능 ID 하나로 독립 검증한다.
- 고위험은 구현자 검증과 구분되는 독립 검증이 필요하다. 별도 QA, 독립된 CI 환경 또는 사용자가 승인한 동등한 검증 경로를 사용할 수 있다.
- 인증·인가, 결제, 개인정보, 데이터 손실, migration·복구 위험은 자동 검증만으로 사람의 최종 판단을 대체하지 않는다.
- 모든 일반·고위험 보고서는 QA 필요 여부와 QA 문서 경로 또는 생략 사유를 남긴다.

## 검증 기준

| 등급 | 최소 검증 |
| --- | --- |
| 경량 | 변경 파일 검사와 가장 작은 관련 lint·test·validator |
| 일반 | 변경 전 기준, 관련 compile·lint·test, 인수 조건 회귀, 최종 기능군 QA 판단 |
| 고위험 | 명시적 승인 확인, 적용 전 기준, 적용 후 동일·확장 검증, 독립 검증, 복구·롤백 검증 또는 안전한 증거 |

validator는 작업 등급, 산출물 존재, 필수 heading과 비어 있지 않은 증거처럼 구조적으로 판별 가능한 항목만 차단한다. 위험의 의미, QA 충분성, 승인 품질, 롤백의 실제 안전성은 자동 판정하지 않고 사용자와 Tech Lead가 검토한다.

등급별 validator 동작은 다음과 같다.

- 경량: 보고서와 인수인계가 없어도 통과한다. 존재하는 산출물은 구조를 검사한다.
- 일반: 보고서가 필요하다. 인수인계가 없으면 보고서에 명시적 생략 사유가 필요하다.
- 고위험: 일반 조건과 고위험 필수 증거 섹션을 모두 검사한다.
- 등급 없음: 과거 산출물 호환을 위해 일반 규칙을 적용하고 경고한다.

프롬프트 길이, 표현 취향, 위험 판단의 의미는 CI 실패 조건으로 만들지 않는다.

## Delta-only Codex 명세

Codex 프롬프트에는 현재 작업에서 달라지는 내용만 적는다. 최소 구성은 작업 ID와 등급, 역할, 목표, 승인된 차이, 포함·제외 범위, 완료·검증 조건, 작업별 중단 조건이다.

다음 내용은 경로를 참조하고 본문에 반복하지 않는다.

- `AGENTS.md` 전체
- 역할 설명과 Skill 원문
- 공통 Git·PR 절차
- 승인 문서 원문
- 보고서·인수인계·PR template 원문

권장 길이는 다음과 같으며 경고 기준일 뿐 CI 차단 조건이 아니다.

| 프롬프트 | 권장 길이 |
| --- | --- |
| 경량 | 300~700자 |
| 일반 | 700~1,400자 |
| 고위험 | 1,200~2,400자 |
| 후속 수정 | 150~500자 |

범위를 설명하는 데 더 긴 내용이 필요하면 정확성을 우선한다. 반복된 공통 규칙과 승인 원문을 제거해 길이를 줄이며, 위험·중단 조건을 생략해서 맞추지 않는다.

## 비소급 전환과 롤백

- 기존 작업 ID 디렉터리, 과거 보고서와 인수인계는 그대로 둔다.
- 새 작업은 PR과 보고서 template의 `작업 등급` 필드를 채운다.
- 등급 없는 기존 PR을 다시 검증할 때는 하위 호환 일반 규칙과 경고를 사용한다.
- 이 하네스 변경을 되돌려야 하면 `HARNESS-LEAN-001` 변경 커밋을 일반 revert로 되돌린다. reset, rebase, force push 또는 과거 산출물 수정은 필요하지 않다.
