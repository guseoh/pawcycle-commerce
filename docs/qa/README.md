# QA 문서(QA Documents)

이 디렉터리는 검증 계획, 테스트 케이스, 버그 리포트, 재검증 결과를 보관한다.

사용 범위는 다음과 같다.

- 테스트 계획(Test Plan)
- 인수 조건(Acceptance Criteria) 매핑
- 수동 테스트 케이스(Manual Test Case)
- 자동화 테스트(Automated Test) 메모
- 버그 리포트(Bug Report)
- 재검증 기록(Retest Record)
- 회귀 위험(Regression Risk) 메모

QA는 결함을 숨기거나 직접 고치기 위해 제품 코드를 변경하지 않는다.

## QA 독립 검증 필요 여부

작업 등급별 QA 시점과 독립 검증의 상세 권위 원본은 `docs/runbook/lean-harness.md`다. 일반 작업은 역할마다 형식적 QA를 반복하지 않고 사용자 목적이 완성되는 마지막 의미 있는 변경에서 필요한 QA를 수행한다. 고위험 작업은 구현자 검증과 구분되는 독립 검증을 유지한다.

기능 구현 PR과 사용자-facing 동작 변경은 PR 또는 작업 보고서에 다음 중 하나를 반드시 남긴다.

- QA 독립 검증 필요: `docs/qa/<작업 ID>/test-plan.md` 작성
- 수정 후 재검증 필요: `docs/qa/<작업 ID>/retest-result.md` 작성
- QA 문서 불필요: PR 또는 보고서에 생략 사유 기록

다음 변경은 QA 독립 검증 대상으로 분류한다.

- 인증/인가
- 결제
- 주문 상태 변경
- 정기배송 상태 전이
- 개인정보
- 재고 부족 처리
- 중복 요청/멱등성
- 데이터 손실 가능성이 있는 변경

문서, 하네스, 보고서처럼 제품 동작을 바꾸지 않는 작업은 QA 문서가 불필요할 수 있다. 이 경우에도 생략 사유를 남긴다.

## 최소 테스트 계획 구조

```markdown
# [작업 ID] 테스트 계획

## 승인된 입력

## 인수 조건 매핑

## 정상 흐름 테스트

## 예외 흐름 테스트

## 경계값 테스트

## 권한 테스트

## 중복 요청 테스트

## 상태 전이 테스트

## 회귀 위험

## 차단 사유
```

## 최소 재검증 결과 구조

```markdown
# [작업 ID] 재검증 결과

## 원 결함 또는 수정 대상

## 재검증 입력

## 재검증 절차

## 기대 결과

## 실제 결과

## 증거

## 남은 위험

## 차단 사유
```
