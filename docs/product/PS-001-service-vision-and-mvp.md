## 3. 서비스 한 문장 정의

PawCycle Commerce는 개와 고양이를 키우는 반려인이 반복 구매하는 소모품을 일반 구매하거나 정기배송 구독으로 관리해, 필요한 시점에 필요한 상품을 놓치지 않고 받을 수 있게 돕는 이커머스(E-commerce) 서비스다.

## 4. 해결하려는 사용자 문제

주요 문제는 반복 구매 자체가 아니라, 반복 구매를 사용자가 계속 기억하고 다시 실행해야 한다는 점이다.

- 소모품이 떨어질 때마다 같은 상품을 다시 찾아 주문해야 한다.
- 사료, 고양이 모래, 배변 패드처럼 사용량이 예측되는 상품도 구매 시점을 놓칠 수 있다.
- 상품마다 소진 속도와 필요한 배송 주기가 달라 사용자가 따로 관리해야 한다.
- 일반 쇼핑몰의 정기배송은 일정, 상품, 수량을 유연하게 바꾸기 어렵다는 인식이 있다.
- 재고 부족, 결제 실패, 배송 일정 변경 같은 예외 상황에서 사용자가 어떤 상태인지 이해하기 어렵다.

## 5. 대상 사용자

### 주요 사용자

- 개 또는 고양이를 키우는 반려인
- 사료, 간식, 고양이 모래, 배변 패드 등 소모품을 반복 구매하는 사용자
- 구매 시기를 놓치면 생활 불편이 바로 발생하는 상품을 관리해야 하는 사용자
- 정기배송을 쓰고 싶지만 일정과 상품 변경의 유연성을 중요하게 보는 사용자

### 보조 사용자

- 상품을 등록하고 재고와 주문 상태를 확인해야 하는 단일 판매자 또는 운영자
- 결제 실패, 재고 부족, 정기 주문 생성 실패를 확인해야 하는 운영 담당자

## 6. 서비스 핵심 가치

- 반복 구매를 자동화해 사용자의 기억 부담을 줄인다.
- 일반 구매와 정기배송을 함께 제공해 사용자가 상품 특성에 따라 구매 방식을 선택할 수 있다.
- 정기배송의 일정, 상품, 수량을 사용자가 이해 가능한 규칙 안에서 조정할 수 있다.
- 주문, 결제, 재고, 구독 상태를 명확히 보여 예외 상황에서도 사용자가 다음 행동을 알 수 있다.
- 기능 구현 이후 성능, 관측성(Observability), 운영 개선까지 학습할 수 있는 실제 업무 흐름을 제공한다.

## 7. 서비스 비전

PawCycle Commerce의 비전은 반려동물 소모품 구매를 일회성 쇼핑이 아니라 생활 주기에 맞춘 관리 경험으로 바꾸는 것이다.

초기에는 단일 판매자, 가상 결제(Virtual Payment), 제한된 상품군으로 시작한다. 이후 실제 결제, 재고 운영, 알림, 성능 측정, 장애 대응, 관측성으로 확장할 수 있도록 제품 흐름과 도메인 정책을 명확히 축적한다.

## 8. 현재 확정된 방향

다음은 현재 작업 입력에서 기본 방향으로 제공된 내용이다. 세부 정책은 별도 Product Decision으로 승인되어야 한다.

- 개와 고양이용 소모품을 주요 대상으로 한다.
- 일반 구매와 정기배송을 모두 지원한다.
- 초기에는 단일 판매자가 상품을 판매한다.
- 초기 결제는 실제 PG(Payment Gateway) 연동이 아니라 가상 결제로 시작한다.
- 백엔드는 Spring Boot를 사용한다.
- 프론트엔드는 Next.js와 TypeScript를 사용한다.
- 데이터베이스는 MySQL을 사용한다.
- 하나의 저장소에서 백엔드, 프론트엔드, 인프라와 문서를 관리한다.
- 구현 이후 인프라, 관측성, 성능 테스트, 성능 개선으로 확장한다.
- AI 에이전트를 실제 개발팀처럼 역할별로 운영한다.
- 사용자는 Product Owner이자 Tech Lead로서 제품 정책과 기술 결정을 최종 승인한다.

## 9. 핵심 사용자 여정

### 여정 A: 일반 구매

1. 사용자가 상품 목록을 탐색한다.
2. 상품 상세에서 가격, 재고 가능 여부, 구매 방식을 확인한다.
3. 상품과 수량을 장바구니에 담는다.
4. 배송 정보를 입력하거나 확인한다.
5. 가상 결제를 수행한다.
6. 주문 생성 결과와 주문 상태를 확인한다.

### 여정 B: 정기배송 신청

1. 사용자가 정기배송 가능한 상품을 확인한다.
2. 상품, 수량, 배송 주기, 첫 배송 희망 시점을 선택한다.
3. 구독 신청 전 예상 다음 배송일과 결제 방식을 확인한다.
4. 가상 결제를 통해 첫 주문 또는 구독 시작 조건을 확인한다.
5. 구독 상태와 다음 배송 예정일을 확인한다.

### 여정 C: 정기배송 관리

1. 사용자가 내 구독 목록을 확인한다.
2. 다음 배송 예정일과 구독 상품을 확인한다.
3. 허용된 마감 시점 이전에 상품, 수량, 배송일, 건너뛰기, 일시정지, 재개, 해지를 요청한다.
4. 변경 결과와 다음 적용 회차를 확인한다.

### 여정 D: 예외 상황 확인

1. 정기 주문 생성 시 재고 부족 또는 가상 결제 실패가 발생한다.
2. 사용자는 구독 또는 주문 상태에서 실패 사유와 다음 조치를 확인한다.
3. 허용된 경우 재시도, 상품 변경, 배송일 변경, 해지를 선택한다.

## 10. MVP Core

MVP Core는 일반 구매와 정기배송의 핵심 차이를 검증하는 최소 기능이다.

- 상품 목록과 상세 조회
- 장바구니
- 일반 주문 생성
- 가상 결제 성공 처리
- 정기배송 신청
- 배송 주기 선택
- 다음 배송 예정일 계산 결과 표시
- 내 구독 목록과 구독 상세 조회
- 한 회차 건너뛰기
- 구독 해지
- 관리자 상품 등록과 재고 수량 관리의 최소 기능

## 11. MVP Support

MVP Support는 핵심 여정을 완성하거나 운영자가 상태를 확인하는 데 필요한 보조 기능이다.

- 회원 가입과 로그인의 최소 기능
- 배송지 입력과 기본 배송지 사용
- 주문 목록과 주문 상세 조회
- 구독 상태 표시
- 가상 결제 실패 모사
- 결제 실패 상태 기록
- 재고 부족 상태 기록
- 운영자가 주문, 구독, 실패 상태를 확인하는 최소 관리자 화면
- QA가 확인할 수 있는 테스트 데이터 생성 방식

## 12. Post-MVP 및 제외 범위

다음 기능은 핵심 가치 검증 이후로 미룬다.

- 실제 PG(Payment Gateway) 연동
- 복수 판매자(Multi-seller) 구조
- 반려동물 프로필 기반 추천
- 상품 검색 고도화
- 쿠폰, 포인트, 등급 할인
- 리뷰, 평점, 문의
- 알림 자동 발송
- 구독 상품 변경의 복잡한 정책
- 구독 일시정지와 재개의 전체 정책
- 배송 추적
- 환불과 교환
- Redis, Kafka, RabbitMQ, Elasticsearch 도입
- 마이크로서비스(Microservices) 분리
- Kubernetes와 구체적 클라우드 인프라
- 구체적인 TPS, 동시 사용자 수, 가용성 수치 목표

제외 이유는 핵심 사용자 문제 검증에 직접 필요하지 않거나, 먼저 도메인 정책과 상태 전이를 학습한 뒤 확장해야 하기 때문이다.

## 14. 주요 사용자 스토리

### US-001: 일반 구매

반려동물 소모품을 지금 바로 구매하려는 사용자로서, 원하는 상품과 수량을 선택하고 가상 결제로 주문을 완료하고 싶다. 그래야 소모품이 부족한 상황에서 즉시 구매할 수 있다.

인수 조건:

- 사용자는 상품 목록과 상세를 볼 수 있다.
- 사용자는 상품과 수량을 장바구니에 담을 수 있다.
- 사용자는 배송 정보를 입력하거나 확인할 수 있다.
- 가상 결제 성공 시 주문이 생성되고 주문 상태를 확인할 수 있다.
- 재고 부족이나 결제 실패 시 주문이 성공한 것처럼 표시되지 않는다.

### US-002: 정기배송 신청

반복 구매가 필요한 상품을 사용하는 반려인으로서, 상품과 수량, 배송 주기를 선택해 정기배송을 신청하고 싶다. 그래야 매번 다시 주문하지 않아도 필요한 시점에 상품을 받을 수 있다.

인수 조건:

- 사용자는 정기배송 가능한 상품을 확인할 수 있다.
- 사용자는 상품, 수량, 배송 주기를 선택할 수 있다.
- 사용자는 다음 배송 예정일을 확인할 수 있다.
- 구독 신청 성공 후 구독 상태와 다음 배송 예정일을 확인할 수 있다.
- 승인되지 않은 배송 주기나 수량은 신청할 수 없다.

### US-003: 한 회차 건너뛰기

소모품이 아직 충분히 남은 사용자로서, 다음 정기배송 한 회차를 건너뛰고 싶다. 그래야 불필요한 재고와 지출을 줄일 수 있다.

인수 조건:

- 사용자는 내 구독 상세에서 다음 배송 예정일을 확인할 수 있다.
- 허용된 조건에서 한 회차 건너뛰기를 요청할 수 있다.
- 건너뛰기 성공 시 다음 배송 예정일이 변경되었음을 확인할 수 있다.
- 이미 처리 중인 회차는 건너뛰기 성공으로 표시되지 않는다.

### US-004: 구독 해지

정기배송이 더 이상 필요 없는 사용자로서, 구독을 해지하고 싶다. 그래야 이후 정기 주문과 결제가 생성되지 않는다.

인수 조건:

- 사용자는 구독 상세에서 해지를 요청할 수 있다.
- 해지 적용 시점이 사용자에게 표시된다.
- 해지된 구독은 이후 정기 주문 생성 대상이 되지 않는다.
- 이미 생성된 주문이 있다면 그 주문의 처리 여부는 별도 정책에 따른다.

### US-005: 운영자의 최소 상품·재고 관리

단일 판매자 운영자로서, 판매할 상품과 재고 수량을 최소 수준으로 관리하고 싶다. 그래야 사용자 주문과 구독 신청을 검증할 수 있다.

인수 조건:

- 운영자는 상품을 등록하거나 판매 가능 상태를 확인할 수 있다.
- 운영자는 재고 수량을 확인하고 조정할 수 있다.
- 재고 부족 상태가 주문 또는 구독 흐름에서 드러난다.

## 15. 정상 흐름

### 일반 주문 정상 흐름

1. 사용자가 상품 목록을 조회한다.
2. 사용자가 상품 상세를 확인한다.
3. 사용자가 상품과 수량을 장바구니에 담는다.
4. 사용자가 배송 정보를 입력한다.
5. 사용자가 가상 결제를 성공 처리한다.
6. 주문이 생성된다.
7. 주문 상태가 사용자에게 표시된다.
8. 필요한 경우 재고가 차감된다.

### 구독 신청 정상 흐름

1. 사용자가 정기배송 가능한 상품을 확인한다.
2. 사용자가 상품, 수량, 배송 주기를 선택한다.
3. 시스템은 다음 배송 예정일을 계산해 표시한다.
4. 사용자가 신청 내용을 확인한다.
5. 가상 결제가 성공하거나 신청 조건이 충족된다.
6. 구독이 활성 상태로 생성된다.
7. 사용자는 구독 상세에서 다음 배송 예정일을 확인한다.

### 한 회차 건너뛰기 정상 흐름

1. 사용자가 내 구독 상세를 연다.
2. 시스템은 다음 배송 예정일과 현재 상태를 표시한다.
3. 사용자가 한 회차 건너뛰기를 요청한다.
4. 시스템은 허용 조건을 확인한다.
5. 다음 배송 예정일이 다음 회차로 이동한다.
6. 사용자는 변경 결과를 확인한다.

## 16. 주요 예외 흐름

- 상품 재고가 부족하면 일반 주문 또는 정기 주문이 성공 상태로 확정되지 않는다.
- 가상 결제가 실패하면 주문 또는 구독 신청은 실패 또는 결제 대기 상태로 표시된다.
- 중복 요청이 들어와도 주문, 결제, 건너뛰기, 해지가 중복으로 적용되어서는 안 된다.
- 이미 처리 중인 정기 주문 회차는 사용자가 변경하거나 건너뛰지 못할 수 있다.
- 해지된 구독은 정기 주문 생성 대상이 아니어야 한다.
- 비활성 상품은 신규 구독 신청 대상에서 제외될 수 있다.
- 승인되지 않은 배송 주기는 선택할 수 없어야 한다.

## 17. 초기 비즈니스 규칙

### 현재 문서에서 확정할 수 있는 원칙

- 일반 구매와 정기배송은 모두 지원한다.
- 초기 판매자는 단일 판매자다.
- 초기 결제는 가상 결제다.
- 백엔드가 가격, 재고, 구독 가능 여부, 주문 상태의 기준이다.
- 프론트엔드는 서버가 제공한 결과를 표시하고 정책을 중복 구현하지 않는다.
- Product Decision은 사용자 승인 전까지 확정되지 않는다.

### 제안 도메인 용어

- 일반 주문(One-time Order): 사용자가 한 번의 구매 의사로 생성하는 주문
- 구독 회차(Subscription Cycle Instance): 특정 구독에서 특정 배송 예정일에 해당하는 회차
- 구독 주문(Subscription Order): 구독 회차에서 생성되는 정기 주문
- 가상 결제(Virtual Payment): 실제 PG 없이 성공과 실패 결과를 모사하는 결제
- 변경 마감 시점(Change Cutoff): 정기 주문 생성 또는 배송 준비 전 변경 요청을 허용하는 마지막 시점

### 제안 도메인 규칙

- 일반 주문은 사용자의 즉시 구매 의사를 처리한다.
- 구독은 반복 주문 생성의 기준 정보를 보관한다.
- 정기 주문은 특정 회차에 대해 생성된 주문이며, 구독 자체와 구분한다.
- 주문 생성, 결제, 건너뛰기, 해지 요청은 중복 적용을 허용하지 않는다.
- 구독 변경은 적용 회차를 명확히 표시해야 한다.

### 사용자 승인이 필요한 정책

- 지원 배송 주기
- 첫 배송일 결정 방식
- 변경 마감 시점
- 건너뛰기 제한
- 일시정지와 재개 정책
- 해지 적용 시점
- 가격 변경 적용 방식
- 재고 부족 처리
- 결제 실패 재시도 정책

### 후속 도메인 설계에서 검토할 질문

- 구독과 정기 주문의 상태 값은 어떻게 구분할 것인가?
- 주문 스냅숏(Order Snapshot)에 어떤 필드를 포함할 것인가?
- 결제 실패 후 구독은 어떤 상태로 남을 것인가?
- 재고 부족이 발생한 회차는 자동 재시도 대상인가?
- 사용자가 같은 요청을 여러 번 보냈을 때 동일 결과를 어떻게 보여줄 것인가?

## 18. Product Decision

### PD-001: MVP에서 일반 구매와 정기배송을 모두 포함할 것인가?

#### 결정할 질문

MVP에서 일반 구매와 정기배송 신청을 모두 완성할지, 정기배송만 먼저 검증할지 결정해야 한다.

#### 필요한 이유

일반 구매는 주문, 결제, 재고의 기본 흐름을 제공하고, 정기배송은 서비스 차별화의 핵심이다. 둘의 범위는 MVP 크기와 도메인 학습 순서에 영향을 준다.

#### 선택지

1. 일반 구매와 정기배송 신청을 모두 포함한다.
2. 일반 구매를 먼저 완성하고 정기배송은 Post-MVP로 미룬다.
3. 정기배송 신청만 먼저 만들고 일반 구매는 최소화한다.

#### 선택지별 영향

| 선택지 | 사용자 경험 | 비즈니스 규칙 | 도메인 복잡도 | 운영 및 예외 처리 | MVP 영향 | 이후 변경 난이도 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 서비스 차별화와 기본 구매를 모두 체험한다. | 주문과 구독 규칙을 함께 정의해야 한다. | 중간 | 결제, 재고, 구독 예외가 함께 필요하다. | MVP가 다소 커진다. | 균형 있게 확장하기 쉽다. |
| 2 | 일반 쇼핑몰과 차이가 약하다. | 주문 규칙에 집중한다. | 낮음 | 예외 범위가 작다. | MVP는 작아진다. | 정기배송을 나중에 붙일 때 모델 조정 가능성이 있다. |
| 3 | 핵심 차별화는 보이나 즉시 구매가 약하다. | 구독 규칙에 집중한다. | 중간 | 정기 주문 전 기본 주문 흐름이 빈약할 수 있다. | MVP는 중간이다. | 일반 주문을 나중에 추가할 때 결제·재고 흐름을 다시 정리해야 한다. |

#### MVP 추천안

선택지 1을 추천한다. 단, 정기배송 관리는 신청, 조회, 한 회차 건너뛰기, 해지로 제한한다.

| 3 | 실제 운영에 가깝다. | 재시도 횟수, 간격, 종료 상태가 필요하다. | 높음 | 배치나 스케줄링 논의가 생긴다. | MVP에 과하다. | 정책을 바꾸기 어렵다. |

### PD-006: 배송 주기는 어떤 방식으로 선택하는가?

#### 결정할 질문

사용자가 선택할 수 있는 배송 주기(Delivery Cycle)를 자유 입력으로 둘지, 제한된 옵션으로 둘지 결정해야 한다.

#### 필요한 이유

배송 주기는 다음 배송일 계산, 구독 변경, 정기 주문 생성, UI 표시, QA 테스트에 영향을 준다.

#### 선택지

1. 2주, 4주, 8주처럼 제한된 옵션을 제공한다.
2. 1주 단위로 사용자가 선택하게 한다.
3. 날짜 기반으로 사용자가 자유롭게 지정하게 한다.

#### 선택지별 영향

| 선택지 | 사용자 경험 | 비즈니스 규칙 | 도메인 복잡도 | 운영 및 예외 처리 | MVP 영향 | 이후 변경 난이도 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 선택이 쉽고 예측 가능하다. | 계산 규칙이 단순하다. | 낮음 | 테스트가 쉽다. | MVP에 적합하다. | 옵션 추가가 쉽다. |
| 2 | 유연성이 높다. | 허용 범위와 계산 규칙이 필요하다. | 중간 | 잘못된 선택 방지가 필요하다. | 약간 커진다. | 이후 제한 옵션으로 줄이면 사용자 영향이 있다. |
| 3 | 가장 유연하다. | 날짜 계산과 예외가 복잡하다. | 높음 | 공휴일, 말일, 배송 불가일 논의가 필요하다. | MVP에 과하다. | 정책 변경 영향이 크다. |

#### MVP 추천안

선택지 1을 추천한다. 정확한 옵션 값은 사용자 승인이 필요하다.

### PD-007: 첫 배송일은 어떻게 정하는가?

#### 결정할 질문

정기배송 신청 시 첫 배송일을 즉시 주문일 기준으로 자동 설정할지, 사용자가 선택할지 결정해야 한다.

#### 필요한 이유

첫 배송일은 구독 시작 판단, 첫 결제, 첫 주문 생성, 다음 배송일 계산에 영향을 준다.

#### 선택지

1. 신청일 기준으로 가장 빠른 배송 가능일을 자동 설정한다.
2. 사용자가 제한된 날짜 범위 안에서 선택한다.
3. 첫 회차는 일반 주문처럼 즉시 생성하고 두 번째 회차부터 주기를 적용한다.

#### 선택지별 영향

| 선택지 | 사용자 경험 | 비즈니스 규칙 | 도메인 복잡도 | 운영 및 예외 처리 | MVP 영향 | 이후 변경 난이도 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 단순하지만 사용자의 일정 통제권이 낮다. | 계산이 단순하다. | 낮음 | 예외가 적다. | 작다. | 날짜 선택 추가가 가능하다. |
| 2 | 사용자가 원하는 시작일을 고를 수 있다. | 선택 가능 범위가 필요하다. | 중간 | 배송 불가일 논의가 생길 수 있다. | 중간이다. | 유연하게 확장 가능하다. |
| 3 | 즉시 필요와 반복 필요를 함께 만족한다. | 첫 주문과 구독 기준 정보가 분리된다. | 중간 | 결제와 주문 상태가 늘어난다. | 중간이다. | 실제 커머스 흐름에 가깝다. |

#### MVP 추천안

선택지 3을 추천한다. 일반 주문과 정기배송의 관계를 학습하기 좋고 사용자가 첫 배송을 기다리는 혼란을 줄인다.

### PD-008: 상품 가격 변경 시 기존 구독에는 어떤 가격을 적용하는가?

#### 결정할 질문

상품 가격이 변경되었을 때 기존 구독의 다음 정기 주문에 최신 가격을 적용할지, 구독 신청 시 가격을 유지할지 결정해야 한다.

#### 필요한 이유

가격 정책은 사용자 신뢰, 주문 스냅숏, 결제 금액, 알림, 약관 설명에 영향을 준다.

#### 선택지

1. 정기 주문 생성 시점의 최신 가격을 적용한다.
2. 구독 신청 시점의 가격을 유지한다.
3. 가격 변경 후 사용자 동의가 있을 때만 다음 회차에 적용한다.

#### 선택지별 영향

| 선택지 | 사용자 경험 | 비즈니스 규칙 | 도메인 복잡도 | 운영 및 예외 처리 | MVP 영향 | 이후 변경 난이도 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 가격 변동이 직관적이지 않을 수 있다. | 최신 가격 기준이 단순하다. | 중간 | 사전 고지 필요성이 생긴다. | 적당하다. | 고지/동의 정책 추가 가능하다. |
| 2 | 사용자에게 예측 가능하다. | 구독 가격 보존 규칙이 필요하다. | 중간 | 원가 상승과 운영 부담이 있다. | 중간이다. | 나중에 최신가 전환 시 사용자 영향이 크다. |
| 3 | 신뢰도가 높다. | 동의 상태와 미동의 처리 규칙이 필요하다. | 높음 | 알림과 대기 상태가 필요하다. | MVP에 과하다. | 가장 유연하지만 복잡하다. |

#### MVP 추천안

선택지 1을 추천하되, MVP에서는 가격 변경 기능 자체를 관리자 범위에서 제한하고 “정기 주문 생성 시점 가격을 주문에 스냅숏으로 저장한다”는 수준으로 둔다.

### PD-009: 정기 주문 생성 시 재고가 부족하면 어떻게 처리하는가?

#### 결정할 질문

#### 필요한 이유

#### 선택지

1. 해당 회차 정기 주문 생성을 실패 상태로 기록한다.
2. 재고 대기 상태로 두고 운영자 또는 시스템 재시도를 기다린다.

| 3 | 일부 상품은 받을 수 있다. | 부분 주문과 금액 계산이 필요하다. | 높음 | CS와 환불 복잡도가 크다. | MVP에 과하다. | 정책 변경이 어렵다. |

### PD-010: 중복 요청은 제품 관점에서 어떤 결과여야 하는가?

주문 생성, 결제, 구독 신청, 건너뛰기, 해지 요청이 중복 전달될 때 사용자에게 어떤 결과를 보장할지 결정해야 한다.

#### 필요한 이유

중복 요청은 중복 주문, 중복 결제, 중복 건너뛰기처럼 사용자 피해를 만들 수 있다.

#### 선택지

1. 동일 요청은 한 번만 적용된 것으로 취급한다.
2. 중복 요청을 오류로 표시하고 사용자가 다시 확인하게 한다.
3. 중복 요청 방지는 MVP에서 명시하지 않는다.

#### 선택지별 영향

| 선택지 | 사용자 경험 | 비즈니스 규칙 | 도메인 복잡도 | 운영 및 예외 처리 | MVP 영향 | 이후 변경 난이도 |
| --- | --- | --- | --- | --- | --- | --- |

| 2 | 사용자가 혼란스러울 수 있다. | 오류 기준이 필요하다. | 낮음 | 지원 문의가 늘 수 있다. | 작다. | 안전성 개선이 나중에 필요하다. |
| 3 | 구현은 단순하다. | 정책이 비어 있다. | 낮음 | 사용자 피해 위험이 있다. | MVP 리스크가 크다. | 나중에 데이터 정리가 어려울 수 있다. |

#### MVP 추천안

## 19. Technical Decision 후보

다음 항목은 현재 확정하지 않는다. 후속 도메인 설계, API 설계, 구현, 측정 결과를 근거로 결정한다.

- API 계약(OpenAPI Contract) 구조
- 데이터베이스 스키마(Database Schema)
- 주문, 결제, 구독, 정기 주문의 상태 모델(State Model)
- 주문 스냅숏(Order Snapshot) 필드
- 멱등성(Idempotency) 보장 방식
- 동시성 제어(Concurrency Control) 방식
- 정기 주문 생성 방식: 스케줄러(Scheduler), 배치(Batch), 수동 트리거 중 선택
- 가상 결제 실패 재처리 방식
- 캐시(Cache) 도입 여부
- 메시지 브로커(Message Broker) 도입 여부
- 검색 엔진(Search Engine) 도입 여부
- 배포 구조와 인프라 구성
- 관측성 도구와 메트릭 체계
- 성능 테스트 도구와 시나리오 구체화

## 21. 초기 워크로드 및 확장성 가설

### 조회 요청이 상대적으로 많은 기능

- 가설: 상품 목록, 상품 상세, 구독 상세, 주문 조회는 쓰기보다 조회가 많을 가능성이 높다.
- 제품·업무 근거: 사용자는 구매 전 여러 상품과 상태를 확인하지만 실제 주문이나 변경은 상대적으로 적다.
- 이후 확인 방법: 접근 로그와 API 요청 수를 기능별로 집계한다.
- 성능 테스트 또는 운영 지표 후보: 응답 시간, 요청 수, 캐시 적중률 후보, 오류율
- 가설이 틀렸을 때 영향: 조회 최적화보다 쓰기 정합성과 트랜잭션 설계가 더 중요해질 수 있다.

### 쓰기와 동시성이 중요한 기능

- 가설: 주문 생성, 재고 차감, 결제, 구독 신청, 건너뛰기, 해지는 중복 실행을 허용할 수 없다.
- 제품·업무 근거: 중복 주문, 초과 재고 차감, 중복 결제, 중복 건너뛰기는 사용자 피해를 만든다.
- 이후 확인 방법: 중복 요청 테스트와 상태 전이 테스트를 작성한다.
- 성능 테스트 또는 운영 지표 후보: 중복 요청 수, 충돌 수, 실패율, 상태 불일치 건수
- 가설이 틀렸을 때 영향: 멱등성 정책 범위를 다시 조정해야 한다.

### 특정 시간에 작업이 집중될 수 있는 기능

- 가설: 정기 주문 생성은 새벽 또는 특정 배치 시간대에 작업이 집중될 수 있다.
- 제품·업무 근거: 구독 회차는 배송 예정일 기준으로 한 번에 처리될 가능성이 있다.
- 이후 확인 방법: 구독 수와 배송 예정일 분포를 수집한다.
- 성능 테스트 또는 운영 지표 후보: 처리 건수, 처리 시간, 실패율, 재처리 수
- 가설이 틀렸을 때 영향: 정기 작업 분산 설계의 우선순위가 낮아질 수 있다.

### 행사나 알림으로 순간 요청 증가가 가능한 기능

- 가설: 인기 상품, 행사 상품, 알림 발송 이후 상품 상세와 주문 요청이 순간 증가할 수 있다.
- 제품·업무 근거: 소모품 할인이나 재입고 알림은 특정 상품으로 사용자를 모을 수 있다.
- 이후 확인 방법: 캠페인 또는 알림 이벤트와 요청량을 비교한다.
- 성능 테스트 또는 운영 지표 후보: 분당 요청 수, 상세 조회 피크, 주문 전환율
- 가설이 틀렸을 때 영향: 피크 대응보다 일반 흐름 안정성에 집중한다.

### 사용자 요청과 무관하게 실행되는 정기 작업

- 가설: 정기 주문 생성, 결제 실패 재처리, 알림 발송은 사용자 클릭 없이 실행될 수 있다.
- 제품·업무 근거: 구독 서비스는 시간 기반 업무가 핵심이다.
- 이후 확인 방법: 정기 작업 실행 로그와 실패 로그를 설계한다.
- 성능 테스트 또는 운영 지표 후보: 작업 성공률, 지연 시간, 재시도 횟수
- 가설이 틀렸을 때 영향: 동기 사용자 요청 중심 설계를 더 오래 유지할 수 있다.

### 데이터가 장기간 누적되는 영역

- 가설: 주문, 결제, 구독 상태 이력, 재고 이력, 알림 이력은 장기간 누적된다.
- 제품·업무 근거: 거래 이력과 상태 변경 이력은 감사와 문제 해결에 필요하다.
- 이후 확인 방법: 테이블별 증가량과 보관 정책을 관찰한다.
- 성능 테스트 또는 운영 지표 후보: 데이터 건수, 조회 시간, 저장소 사용량
- 가설이 틀렸을 때 영향: 보관 정책과 인덱스 검토 시점이 늦어질 수 있다.

### 장애 발생 후 재실행이 필요한 작업

- 가설: 정기 주문 생성, 결제 실패 재처리, 알림 발송은 실패 후 재실행이 필요하다.
- 제품·업무 근거: 사용자의 직접 요청 없이 실행되는 작업은 실패를 사용자가 즉시 복구하기 어렵다.
- 이후 확인 방법: 실패 상태와 재처리 결과를 기록한다.
- 성능 테스트 또는 운영 지표 후보: 재처리 성공률, 최종 실패율, 재처리 지연 시간
- 가설이 틀렸을 때 영향: 재처리 설계보다 사용자 수동 복구 흐름이 중요해질 수 있다.

사용자 스토리

비즈니스 규칙

관련 도메인 용어
→ 구독(Subscription), 배송 주기(Delivery Cycle), 정기 주문(Recurring Order)
이후 검증할 테스트
→ 구현 시 구독 신청 성공, 허용되지 않은 배송 주기 거부, 다음 배송일 표시 테스트

```

사용자 문제

```

### 추적 3: 정기배송 유연성

```text
사용자 문제

사용자 스토리
→ US-003: 한 회차 건너뛰기
비즈니스 규칙

이후 관찰할 운영 또는 성능 지표
→ 관측성 단계에서 건너뛰기 요청 수, 실패율, 마감 이후 요청 수 정의
```

## 23. 위험과 미결정 사항

### 제품 위험

- MVP에 일반 구매와 정기배송을 모두 포함하면 범위가 커질 수 있다.
- 구독 변경 기능을 너무 많이 포함하면 핵심 가치 검증보다 정책 설계가 커질 수 있다.
- 가격, 재고, 결제 실패 정책이 늦게 결정되면 백엔드와 QA 작업이 막힐 수 있다.
- 가상 결제 범위가 불명확하면 실제 PG 도입 전 상태 모델을 다시 바꿀 수 있다.

### 미결정 Product Decision

- MVP 범위 확정
- 정기배송 관리 기능 범위
- 관리자 기능 범위
- 가상 결제 실패와 재시도 범위
- 배송 주기 옵션
- 첫 배송일 정책
- 가격 변경 정책
- 재고 부족 처리 정책
- 중복 요청의 제품 결과

### 미결정 Technical Decision

- API 계약
- DB 스키마
- 상태 모델
- 멱등성 구현 방식
- 동시성 제어 방식
- 정기 작업 실행 방식
- 성능 측정 방식

## 24. 후속 작업 제안

1. `[PS-001][TL] Product Decision 승인`
   - 이 문서의 Product Decision을 사용자가 승인, 수정, 보류한다.
2. `[PS-002][PO] MVP 상세 PRD`
   - 승인된 MVP 범위를 기준으로 기능별 PRD와 인수 조건을 상세화한다.
3. `[ARCH-001][TL] 초기 도메인 모델과 상태 전이 검토`
   - 구독, 주문, 정기 주문, 결제, 재고의 상태 모델을 결정한다.
4. `[PS-003][UX] MVP 사용자 흐름과 화면 목록`
   - 일반 구매, 정기배송 신청, 구독 조회와 건너뛰기 화면 흐름을 설계한다.
5. `[ARCH-002][BE] API 계약 초안`
   - 승인된 정책을 기준으로 OpenAPI 계약 초안을 작성한다.
6. `[QA-001][QA] MVP 인수 테스트 계획`
   - 사용자 스토리와 인수 조건을 테스트 케이스로 연결한다.
7. `[PERF-001][SRE] 성능 가설 검증 계획`
   - 구현 이후 측정할 시나리오와 지표 후보를 정리한다.
