# PS-002 1차 MVP 기능 요구사항

## 문제

PS-001에서 첫 번째 수직 MVP의 제품 범위와 핵심 Product Decision은 확정됐다. 이제 후속 DOMAIN-001, API-001, UX/UI, QA가 같은 범위를 추측 없이 이어받을 수 있도록 기능 요구사항, 검증 가능한 인수 조건과 최소 추적성을 정리한다.

이번 문서는 요구사항 정의 산출물이다. 도메인 상세 설계, API 계약, 데이터베이스 설계와 구현은 확정하지 않는다.

## 대상 사용자

- 로그인한 회원
- 개와 고양이용 사료를 정기배송으로 구독하려는 회원

비회원은 구독 생성과 자신의 구독 조회 기능을 사용할 수 없다.

## 범위

- 상품 목록 조회
- 상품 상세와 SKU 조회
- 구독 생성
- 내 구독 목록 조회
- 내 구독 상세 조회
- 다음 주문 예정일 확인
- 구독 생성·조회 접근 규칙

## 제외 범위

- JPA Entity와 애플리케이션 코드
- ERD와 데이터베이스 스키마
- API 요청·응답 구조
- HTTP 상태와 오류 코드 확정
- 인증·인가 구현 방식
- UI 컴포넌트와 화면 설계
- 정기 주문 자동 생성
- 결제, 재고와 배송
- 구독 변경·건너뛰기·일시정지·재개·해지
- 구독 상태 모델
- 새로운 기술과 의존성 도입
- PS-001에 요구사항 ID 소급 적용
- 요구사항 추적성 전용 플랫폼 도입
- 요구사항 ID 자동 검증 CI 구현

## 승인된 Product Decision

| 항목 | 결정 |
| --- | --- |
| MVP 성격 | 구독 생성·조회 도메인을 검증하는 첫 번째 수직 MVP |
| 구독 단위 | 첫 번째 MVP에서 구독 하나는 SKU 하나만 대상으로 함 |
| 복수 SKU 구독 | Post-MVP |
| 상품 예시와 검증 범위 | 개·고양이용 사료 |
| 배송 주기 | 2주, 4주, 8주 |
| 최초 주문 예정일 | 사용자가 선택하지 않음 |
| 다음 주문 예정일 | 구독 생성일 + 선택한 배송 주기 |
| 날짜 기준 | 날짜 단위, Asia/Seoul |
| 휴일 보정 | 휴일·주말·영업일 보정 없음 |
| 예정일 표시 | 다음 주문 예정일만 제공, 배송 예정일 제외 |
| 정기 주문 자동 생성 | 첫 번째 MVP 제외 |
| 구독 수량 범위 | 1~10 |
| 구독 대상 SKU | 존재하며 구독 가능한 SKU |
| 재고·품절·일반 판매 상태 | 이번 구독 생성 조건에 연결하지 않음 |
| 구독 상태 모델 | 첫 번째 MVP에서 사용하지 않음 |

## 공통 비즈니스 규칙

- 회원은 존재하며 구독 가능한 SKU만 구독할 수 있다.
- 구독 생성 수량은 1~10만 허용한다.
- 배송 주기는 2주, 4주, 8주 중 하나만 허용한다.
- 다음 주문 예정일은 구독 생성일에 선택한 배송 주기를 더해 계산한다.
- 날짜 계산과 표시는 Asia/Seoul 기준 날짜 단위로 다룬다.
- 휴일, 주말과 영업일 보정은 하지 않는다.
- 다음 주문 예정일은 표시용 예정 정보이며 해당 날짜에 실제 주문을 자동 생성하지 않는다.
- 첫 번째 MVP에서는 배송 예정일을 제공하지 않는다.
- 첫 번째 MVP에서는 구독 상태 모델을 사용하지 않는다.
- 재고, 품절과 일반 판매 상태는 이번 구독 생성 성공·실패 조건에 연결하지 않는다.
- 회원은 자신의 구독만 조회할 수 있다.

## 날짜 표시 기준

- 계약에서 사용할 날짜 기준 형식 후보는 `YYYY-MM-DD`다.
- 사용자 화면 표시는 `YYYY. M. D.` 형식을 기준으로 요구사항을 작성한다.
- API 계약의 최종 날짜 표현은 API-001에서 확정한다.

예시는 다음과 같다.

```text
구독 생성일: 2026-07-03
선택한 배송 주기: 4주
다음 주문 예정일: 2026-07-31
사용자 화면 표시 후보: 2026. 7. 31.
```

## 사용자 스토리

- 회원으로서 나는 구독 가능한 사료 상품과 SKU를 확인하고 싶다. 그래야 구독할 상품을 선택할 수 있다.
- 회원으로서 나는 수량과 배송 주기를 선택해 구독을 만들고 싶다. 그래야 반복 구매 일정을 관리할 수 있다.
- 회원으로서 나는 내 구독과 다음 주문 예정일을 확인하고 싶다. 그래야 다음 반복 구매 시점을 알 수 있다.
- 서비스 운영 관점에서 비회원과 다른 회원의 구독 접근은 제한돼야 한다. 그래야 회원별 구독 정보가 분리된다.

## 정상 흐름

```text
회원 로그인
→ 상품 목록 조회
→ 상품 상세와 SKU 조회
→ 구독 가능한 SKU 선택
→ 수량 1~10 입력
→ 배송 주기 2주·4주·8주 중 선택
→ 구독 생성
→ 생성된 구독 상세 확인
→ 내 구독 목록 조회
→ 다음 주문 예정일 확인
```

## 예외 흐름

- 비회원이 구독 생성·조회 화면에 접근하면 로그인 화면으로 이동한다.
- 존재하지 않는 SKU로 구독을 생성하려 하면 구독이 생성되지 않는다.
- 구독 불가능한 SKU로 구독을 생성하려 하면 구독이 생성되지 않는다.
- 수량이 1보다 작거나 10보다 크면 구독이 생성되지 않는다.
- 2주, 4주, 8주가 아닌 배송 주기를 선택하면 구독이 생성되지 않는다.
- 필수 입력이 누락되면 구독이 생성되지 않는다.
- 회원이 다른 회원의 구독을 조회하려 하면 조회할 수 없는 구독으로 처리한다.

HTTP `401`, `404`, 오류 코드와 오류 응답 JSON은 이번 작업에서 확정하지 않고 API-001로 전달한다.

## 기능 요구사항

### REQ-PRODUCT-001 상품 목록 조회

회원은 첫 번째 MVP 검증 범위인 개와 고양이용 사료 상품 목록을 확인할 수 있다.

#### 표시 정보

- 상품 ID
- 상품명
- 대상 동물
- 짧은 설명
- SKU별 표시 가격
- 구독 가능한 SKU 존재 여부

#### 정상 인수 조건

- AC-PRODUCT-001-01: 상품 목록에는 상품 ID, 상품명, 대상 동물, 짧은 설명, SKU별 표시 가격과 구독 가능한 SKU 존재 여부가 표시된다.
- AC-PRODUCT-001-02: 구독 가능한 SKU가 하나 이상 있는 상품은 목록에서 그 존재 여부를 확인할 수 있다.
- AC-PRODUCT-001-03: 첫 번째 MVP의 상품 예시와 검증 데이터는 개와 고양이용 사료를 중심으로 제공된다.

#### 주요 실패 인수 조건

- AC-PRODUCT-001-04: 상품 목록에 간식, 고양이 모래, 배변 패드, 위생용품 검증 데이터가 필수 범위로 포함되지 않는다.

#### 제외 범위

- 정렬, 필터, 검색, 페이지네이션 정책
- 할인 가격, 쿠폰, 포인트 표시
- 재고와 품절 표시 정책

#### 관련 Product Decision

- 첫 번째 MVP 상품 예시와 검증 범위는 개·고양이용 사료다.
- 재고, 품절과 일반 판매 상태는 이번 구독 생성 조건에 연결하지 않는다.

#### 후속 설계로 넘길 내용

- 상품 목록 API의 요청·응답 구조는 API-001에서 정한다.
- 상품과 SKU의 도메인 관계는 DOMAIN-001에서 구체화한다.

### REQ-PRODUCT-002 상품 상세와 SKU 조회

회원은 상품 상세에서 SKU 목록과 SKU별 구독 가능 여부를 확인할 수 있다.

#### 표시 정보

- 상품명
- 상품 설명
- 대상 동물
- SKU 목록
- SKU별 용량 또는 구성
- SKU별 표시 가격
- 구독 가능 여부
- 선택 가능한 배송 주기

#### 정상 인수 조건

- AC-PRODUCT-002-01: 상품 상세에는 상품명, 상품 설명, 대상 동물, SKU 목록이 표시된다.
- AC-PRODUCT-002-02: 각 SKU에는 용량 또는 구성, 표시 가격과 구독 가능 여부가 표시된다.
- AC-PRODUCT-002-03: 상품 상세에는 선택 가능한 배송 주기 2주, 4주, 8주가 표시된다.
- AC-PRODUCT-002-04: 구독 불가능한 SKU는 회원이 구독 대상으로 사용할 수 없음을 알 수 있게 구분된다.

#### 주요 실패 인수 조건

- AC-PRODUCT-002-05: 존재하지 않는 상품이나 SKU는 구독 생성 대상으로 선택할 수 없다.
- AC-PRODUCT-002-06: 배송 예정일은 상품 상세 표시 정보로 제공하지 않는다.

#### 제외 범위

- SKU 코드 생성 규칙
- SKU별 재고 수량과 품절 처리
- 판매 상태 전이
- SKU별 배송 주기 제한 정책

#### 관련 Product Decision

- 회원은 존재하며 구독 가능한 SKU만 구독할 수 있다.
- 배송 주기는 2주, 4주, 8주다.

#### 후속 설계로 넘길 내용

- 구독 가능 여부의 도메인 표현은 DOMAIN-001에서 정한다.
- 상품 상세 API의 오류 처리와 응답 JSON은 API-001에서 정한다.

### REQ-SUB-001 구독 생성

로그인한 회원은 존재하며 구독 가능한 SKU 하나, 수량 1~10과 허용된 배송 주기를 입력해 구독을 생성할 수 있다.

#### 표시 정보

- 선택한 상품명
- 선택한 SKU
- SKU 표시 가격
- 수량
- 배송 주기
- 다음 주문 예정일

#### 정상 인수 조건

- AC-SUB-001-01: 로그인한 회원이 존재하며 구독 가능한 SKU, 수량 1~10과 배송 주기 2주, 4주, 8주 중 하나를 입력하면 구독이 생성된다.
- AC-SUB-001-02: 생성된 구독은 회원, SKU 하나, 수량, 배송 주기, 구독 생성일과 다음 주문 예정일을 가진다.
- AC-SUB-001-03: 구독 생성 후 회원은 생성된 구독 상세를 확인할 수 있다.
- AC-SUB-001-04: 구독 생성 후 회원은 다음 주문 예정일을 확인할 수 있다.

#### 주요 실패 인수 조건

- AC-SUB-001-05: 비로그인 사용자는 구독을 생성할 수 없다.
- AC-SUB-001-06: 존재하지 않는 SKU를 입력하면 구독이 생성되지 않는다.
- AC-SUB-001-07: 구독 불가능한 SKU를 입력하면 구독이 생성되지 않는다.
- AC-SUB-001-08: 수량이 1보다 작거나 10보다 크면 구독이 생성되지 않는다.
- AC-SUB-001-09: 2주, 4주, 8주가 아닌 배송 주기를 입력하면 구독이 생성되지 않는다.
- AC-SUB-001-10: SKU, 수량 또는 배송 주기 중 필수 입력이 누락되면 구독이 생성되지 않는다.

#### 접근 규칙

- 구독 생성은 로그인한 회원만 수행할 수 있다.
- 비회원이 구독 생성 화면에 접근하면 로그인 화면으로 이동한다.

#### 제외 범위

- 여러 SKU를 하나의 구독에 포함하는 기능
- 정기 주문 자동 생성
- 결제, 재고 예약, 재고 차감, 배송 처리
- 구독 상태 모델
- 구독 변경·건너뛰기·일시정지·재개·해지

#### 관련 Product Decision

- 첫 번째 MVP에서 구독 하나는 SKU 하나만 대상으로 한다.
- 구독 수량 범위는 1~10이다.
- 배송 주기는 2주, 4주, 8주다.
- 재고, 품절과 일반 판매 상태는 이번 구독 생성 조건에 연결하지 않는다.

#### 후속 설계로 넘길 내용

- 구독 생성 불변 조건은 DOMAIN-001에서 구체화한다.
- API 요청·응답 구조, HTTP 상태와 오류 코드는 API-001에서 정한다.

### REQ-SUB-002 내 구독 목록 조회

로그인한 회원은 자신이 생성한 구독 목록을 확인할 수 있다.

#### 표시 정보

- 구독 ID
- 상품명
- SKU
- 수량
- 배송 주기
- 다음 주문 예정일

#### 정상 인수 조건

- AC-SUB-002-01: 로그인한 회원은 자신의 구독 목록을 조회할 수 있다.
- AC-SUB-002-02: 내 구독 목록에는 구독 ID, 상품명, SKU, 수량, 배송 주기와 다음 주문 예정일이 표시된다.
- AC-SUB-002-03: 구독 목록의 각 항목은 하나의 SKU만 대상으로 표시된다.
- AC-SUB-002-04: 구독 상태는 첫 번째 MVP의 목록 표시 정보에 포함하지 않는다.

#### 주요 실패 인수 조건

- AC-SUB-002-05: 비로그인 사용자는 내 구독 목록을 조회할 수 없다.
- AC-SUB-002-06: 회원은 다른 회원의 구독을 내 구독 목록에서 볼 수 없다.

#### 접근 규칙

- 내 구독 목록은 로그인한 회원 본인의 구독만 포함한다.
- 비회원이 내 구독 목록 화면에 접근하면 로그인 화면으로 이동한다.

#### 제외 범위

- 구독 상태 필터
- 구독 변경, 건너뛰기, 일시정지, 재개, 해지 액션
- 다음 배송 예정일

#### 관련 Product Decision

- 첫 번째 MVP에서는 구독 상태 모델을 사용하지 않는다.
- 다음 주문 예정일만 제공하며 배송 예정일은 제공하지 않는다.

#### 후속 설계로 넘길 내용

- 회원과 구독의 관계는 DOMAIN-001에서 구체화한다.
- 목록 API의 인증·인가 처리 방식은 API-001에서 정한다.

### REQ-SUB-003 내 구독 상세 조회

로그인한 회원은 자신이 생성한 구독의 상세 정보를 확인할 수 있다.

#### 표시 정보

- 구독 ID
- 상품명
- SKU
- SKU 표시 가격
- 수량
- 배송 주기
- 구독 생성일
- 다음 주문 예정일

#### 정상 인수 조건

- AC-SUB-003-01: 로그인한 회원은 자신의 구독 상세를 조회할 수 있다.
- AC-SUB-003-02: 구독 상세에는 구독 ID, 상품명, SKU, SKU 표시 가격, 수량, 배송 주기, 구독 생성일과 다음 주문 예정일이 표시된다.
- AC-SUB-003-03: 구독 상세의 대상 SKU는 하나만 표시된다.
- AC-SUB-003-04: 구독 상세에는 다음 배송 예정일을 표시하지 않는다.

#### 주요 실패 인수 조건

- AC-SUB-003-05: 비로그인 사용자는 구독 상세를 조회할 수 없다.
- AC-SUB-003-06: 회원이 다른 회원의 구독 상세에 접근하면 조회할 수 없는 구독으로 처리한다.
- AC-SUB-003-07: 존재하지 않는 구독 상세는 조회할 수 없는 구독으로 처리한다.

#### 접근 규칙

- 회원은 자신의 구독만 조회할 수 있다.
- 다른 회원의 구독 접근은 조회할 수 없는 구독으로 처리한다.

#### 제외 범위

- 구독 상태 표시
- 주문 내역, 결제 내역, 배송 내역
- 구독 변경 관련 액션

#### 관련 Product Decision

- 첫 번째 MVP에서 구독 하나는 SKU 하나만 대상으로 한다.
- 회원은 다른 회원의 구독을 조회할 수 없다.

#### 후속 설계로 넘길 내용

- 다른 회원의 구독 접근에 대한 HTTP 상태와 오류 응답은 API-001에서 정한다.
- 구독 식별자와 조회 책임은 DOMAIN-001에서 구체화한다.

### REQ-SUB-004 다음 주문 예정일 확인

회원은 구독 생성 결과, 내 구독 목록과 내 구독 상세에서 다음 주문 예정일을 확인할 수 있다.

#### 표시 정보

- 다음 주문 예정일
- 배송 주기
- 구독 생성일

#### 정상 인수 조건

- AC-SUB-004-01: 다음 주문 예정일은 구독 생성일에 선택한 배송 주기를 더해 산출된다.
- AC-SUB-004-02: 날짜 계산은 Asia/Seoul 기준 날짜 단위로 다룬다.
- AC-SUB-004-03: 휴일, 주말과 영업일 보정을 적용하지 않는다.
- AC-SUB-004-04: 사용자 화면에는 다음 주문 예정일을 `YYYY. M. D.` 형식으로 표시한다.
- AC-SUB-004-05: 구독 생성 결과, 내 구독 목록과 내 구독 상세에서 다음 주문 예정일을 확인할 수 있다.

#### 주요 실패 인수 조건

- AC-SUB-004-06: 최초 주문 예정일은 사용자가 직접 선택할 수 없다.
- AC-SUB-004-07: 다음 배송 예정일은 다음 주문 예정일 대신 표시되지 않는다.
- AC-SUB-004-08: 다음 주문 예정일 도래만으로 실제 정기 주문이 자동 생성되지 않는다.

#### 제외 범위

- 날짜 변경 정책
- 영업일 보정 정책
- 다음 배송 예정일
- 정기 주문 자동 생성

#### 관련 Product Decision

- 다음 주문 예정일은 구독 생성일 + 선택한 배송 주기다.
- 기준 시간대는 Asia/Seoul이다.
- 다음 주문 예정일만 제공하며 배송 예정일은 제공하지 않는다.

#### 후속 설계로 넘길 내용

- 다음 주문 예정일 계산 책임과 날짜 값 객체 필요성은 DOMAIN-001에서 정한다.
- API 계약의 날짜 표현은 API-001에서 확정한다.

### REQ-AUTH-001 인증된 회원만 구독 생성·조회

구독 생성과 내 구독 조회 기능은 인증된 회원만 사용할 수 있다.

#### 정상 인수 조건

- AC-AUTH-001-01: 로그인한 회원은 구독 생성 화면에 접근할 수 있다.
- AC-AUTH-001-02: 로그인한 회원은 내 구독 목록과 상세 화면에 접근할 수 있다.

#### 주요 실패 인수 조건

- AC-AUTH-001-03: 비회원이 구독 생성 화면에 접근하면 로그인 화면으로 이동한다.
- AC-AUTH-001-04: 비회원이 내 구독 목록 화면에 접근하면 로그인 화면으로 이동한다.
- AC-AUTH-001-05: 비회원이 내 구독 상세 화면에 접근하면 로그인 화면으로 이동한다.

#### 제외 범위

- 인증 토큰, 세션, 쿠키, Spring Security 구현 방식
- HTTP 상태와 오류 응답 JSON

#### 관련 Product Decision

- 첫 번째 MVP의 사용자는 로그인한 회원이다.

#### 후속 설계로 넘길 내용

- API 인증 실패 응답은 API-001에서 정한다.
- 인증·인가 구현 방식은 후속 Backend 또는 Security 작업에서 정한다.

### REQ-AUTH-002 다른 회원의 구독 조회 차단

## Product Decision

이번 PS-002에서 새로 확정된 제품 요구사항 입력은 다음과 같다.

- 구독 수량 범위는 1~10이다.
- 상품 목록 표시 정보는 상품 ID, 상품명, 대상 동물, 짧은 설명, SKU별 표시 가격, 구독 가능한 SKU 존재 여부다.
- 상품 상세 표시 정보는 상품명, 상품 설명, 대상 동물, SKU 목록, SKU별 용량 또는 구성, SKU별 표시 가격, 구독 가능 여부, 선택 가능한 배송 주기다.
- 구독 생성 성공 조건은 로그인한 회원, 존재하는 SKU, 구독 가능한 SKU, 수량 1~10, 배송 주기 2주·4주·8주 중 하나다.
- 구독 생성 실패 조건은 비로그인, 존재하지 않는 SKU, 구독 불가능한 SKU, 수량 범위 위반, 허용되지 않은 배송 주기, 필수 입력 누락이다.
- 내 구독 목록과 상세 표시 정보는 본 문서의 요구사항별 표시 정보로 확정한다.
- 비회원은 구독 생성·조회 화면 접근 시 로그인 화면으로 이동한다.
- 다른 회원의 구독 접근은 조회할 수 없는 구독으로 처리한다.
- 첫 번째 MVP에서는 구독 상태 모델을 사용하지 않는다.
