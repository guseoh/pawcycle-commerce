# DOMAIN-001 1차 MVP 구독 도메인 설계

## 작업 목적

PS-001과 PS-002에서 승인된 첫 번째 수직 MVP 요구사항을 바탕으로 회원·고객, 상품, SKU, 구독의 도메인 의미, 책임, 관계, 불변 조건과 날짜 계산 책임을 구체화한다.

이번 문서는 도메인 발견·설계 문서다. JPA Entity, 데이터베이스, API, 패키지 구조와 애플리케이션 코드는 확정하거나 구현하지 않는다.

## 결정 상태 기준

- Approved Input: PS-001과 PS-002에서 이미 승인된 제품 규칙
- Accepted Domain Design: 승인 입력을 구현 가능한 책임과 불변 조건으로 해석한 현재 설계
- Proposed Technical Design: Aggregate, 패키지, 타입과 같은 후속 기술 검토 후보
- Deferred: 첫 번째 MVP에서 다루지 않는 결정

## 논리적 도메인 경계

### 회원 또는 고객 영역

책임은 구독 주체와 소유자 식별이다.

- 인증된 사용자를 회원으로 본다.
- 구독은 소유자를 식별할 수 있어야 한다.
- 회원 식별 방식, 인증 토큰, 세션과 Spring Security 구현은 도메인 경계 밖이다.

회원과 고객은 관점에 따라 구분한다.

- 회원: 로그인과 접근 자격을 가진 사용자라는 제품·인증 관점
- 고객: 상품을 구매하거나 구독하는 커머스 행위자라는 도메인 관점

첫 번째 MVP에서는 구독 생성과 조회 주체가 로그인한 회원이므로, 도메인 문서에서는 "회원"을 주 용어로 사용한다. 커머스 전체 문맥에서 일반 구매와 구독을 모두 포함하는 상위 행위자는 "고객"으로 유지한다.

이 구분은 Accepted Domain Design이다. 회원 등급, 계정 상태, 고객 식별자 구조는 확정하지 않는다.

### 상품 카탈로그 영역

책임은 상품 정보, SKU와 구독 가능 여부를 제공하는 것이다.

- Product는 사용자에게 보여지는 공통 상품 정보다.
- SKU는 용량·구성 등 선택이 완료된 실제 구독 선택 단위다.
- Product는 하나 이상의 SKU를 제공할 수 있다.
- SKU는 자신의 구독 가능 여부를 표현해야 한다.
- 구독 생성에서는 SKU의 구독 가능 여부만 사용한다.
- 재고, 품절과 일반 판매 상태는 이번 불변 조건에 포함하지 않는다.

Product와 SKU를 같은 Aggregate로 둘지, 별도 Aggregate로 둘지는 Proposed Technical Design이다. 첫 번째 MVP에서는 상품 상세 조회와 구독 생성에서 SKU의 구독 가능 여부를 참조해야 하므로, Product가 SKU 목록을 제공하고 구독 생성은 조회된 SKU의 도메인 의미를 사용하는 협력 관계로 설명한다.

JPA 연관관계, FK, 테이블 구조와 조회 쿼리는 확정하지 않는다.

### 구독 영역

책임은 유효한 구독 생성과 자신의 구독 조회에 필요한 도메인 의미를 보존하는 것이다.

Subscription은 첫 번째 MVP에서 다음 내용을 설명한다.

- 구독 소유자
- 대상 SKU 하나
- 구독 수량
- 배송 주기
- 구독 생성일
- 다음 주문 예정일

Subscription은 실제 정기 주문, 결제, 재고 예약, 배송과 구독 변경 흐름을 담당하지 않는다.

## Product와 SKU

### Product

Product는 사용자에게 보여지는 공통 상품 정보다.

첫 번째 MVP의 요구사항에서 Product는 다음 표시 정보를 제공해야 한다.

- 상품 ID
- 상품명
- 대상 동물
- 짧은 설명 또는 상품 설명
- SKU 목록 또는 SKU별 표시 정보
- 구독 가능한 SKU 존재 여부

Product의 책임은 상품 카탈로그 관점의 정보 제공이다. Product는 구독 생성 불변 조건을 직접 검증하지 않는다.

### SKU

SKU는 용량·구성 등 선택이 완료된 실제 구독 선택 단위다.

SKU는 다음 도메인 의미를 가진다.

- Product에 속하거나 연결된 선택 단위
- 용량 또는 구성
- 표시 가격
- 구독 가능 여부

구독 생성에서 SKU는 다음 질문에 답할 수 있어야 한다.

```text
이 SKU를 현재 첫 번째 MVP 구독 대상으로 사용할 수 있는가?
```

구독 가능 여부는 SKU가 제공하는 능력 또는 속성으로 우선 설명한다. 단순 Boolean, enum, 값 객체와 DB 표현은 확정하지 않는다.

재고, 품절, 일반 판매 상태는 별도 제품 개념이다. 첫 번째 MVP의 구독 생성 불변 조건은 SKU의 구독 가능 여부만 사용한다.

## Subscription

Subscription은 회원이 SKU 하나를 정해진 수량과 배송 주기로 반복해서 받기 위해 만든 약속이다.

### 최소 구성

- 소유자: 구독을 만든 회원 또는 고객 식별
- 대상 SKU: 구독 가능한 SKU 하나
- 수량: SubscriptionQuantity
- 배송 주기: DeliveryCycle
- 구독 생성일: Asia/Seoul 기준 날짜
- 다음 주문 예정일: 구독 생성일과 배송 주기로 계산한 날짜

### 불변 조건

- 소유자가 반드시 존재한다.
- 대상 SKU는 하나다.
- SKU는 구독 가능해야 한다.
- 수량은 1~10이다.
- 배송 주기는 2주·4주·8주 중 하나다.
- 생성일과 다음 주문 예정일이 존재한다.
- 다음 주문 예정일은 생성일보다 이후다.
- 다음 주문 예정일은 생성일과 배송 주기로 결정된다.
- 첫 번째 MVP에는 구독 상태가 없다.

### 책임 구분

소유자와 SKU가 저장소에 실제로 존재하는지 조회하는 책임은 애플리케이션 경계에 있다.

전달받은 도메인 값의 규칙을 검증하고 일관된 Subscription을 만드는 책임은 구독 도메인 경계에 있다.

```text
애플리케이션 경계
→ 인증 회원 식별
→ SKU 조회
→ Asia/Seoul 기준 구독 생성일 공급
→ 구독 생성 흐름 조정

도메인 경계
→ SKU 구독 가능 여부 검사
→ 수량 검사
→ 배송 주기 검사
→ 다음 주문 예정일 계산
→ 일관된 Subscription 생성
```

## 값 객체 후보 판단

### SubscriptionQuantity

| 항목 | 내용 |
| --- | --- |
| 해결하는 문제 | 구독 수량 범위가 여러 유스케이스에 흩어지는 문제 |
| 보장하는 불변 조건 | 수량은 1~10 |
| 값 객체 이점 | 생성 시점에 범위를 보장하고 Subscription 생성 규칙을 단순화 |
| 현재 단계의 복잡성 | 수량 연산 정책은 없으므로 범위 보장 외 역할은 작음 |
| 최종 판단 상태 | Accepted |

SubscriptionQuantity는 첫 번째 MVP 구독 생성에서 직접 불변 조건을 보호하므로 값 객체로 다루는 설계를 채택한다.

### DeliveryCycle

| 항목 | 내용 |
| --- | --- |
| 해결하는 문제 | 허용 배송 주기와 다음 날짜 계산 기간이 흩어지는 문제 |
| 보장하는 불변 조건 | 배송 주기는 2주, 4주, 8주 중 하나 |
| 값 객체 이점 | 허용 주기 검증과 날짜 계산에 필요한 기간 제공을 한 의미로 묶음 |
| 현재 단계의 복잡성 | DB 저장 값, enum 이름과 코드 표현은 아직 미정 |
| 최종 판단 상태 | Accepted |

DeliveryCycle은 첫 번째 MVP에서 승인된 고정 선택지를 보호하고 다음 주문 예정일 계산에 사용되므로 값 객체로 다루는 설계를 채택한다.

### SubscriptionCreatedDate

| 항목 | 내용 |
| --- | --- |
| 해결하는 문제 | 날짜가 Asia/Seoul 기준 날짜 단위라는 의미 전달 |
| 보장하는 불변 조건 | 생성일이 존재한다 |
| 값 객체 이점 | 시간대와 날짜 단위 의미를 타입으로 드러낼 수 있음 |
| 현재 단계의 복잡성 | 추가 불변 조건이 없고 API·DB 표현도 미확정 |
| 최종 판단 상태 | Deferred |

현재 단계에서는 `LocalDate`에 대응하는 도메인 날짜 값으로 설명한다. 별도 값 객체는 날짜 변경 정책이나 추가 규칙이 생길 때 재검토한다.

### NextOrderScheduledDate

| 항목 | 내용 |
| --- | --- |
| 해결하는 문제 | 다음 주문 예정일과 배송 예정일 혼동 방지 |
| 보장하는 불변 조건 | 다음 주문 예정일이 존재하고 생성일보다 이후 |
| 값 객체 이점 | 주문 예정일이라는 의미를 타입으로 분리 가능 |
| 현재 단계의 복잡성 | 생성일과 배송 주기에서 계산되는 파생 값이며 별도 행위가 없음 |
| 최종 판단 상태 | Deferred |

현재 단계에서는 Subscription이 계산해 보유하거나 노출하는 날짜 의미로 설명한다. 별도 값 객체는 주문 자동 생성이나 날짜 변경 정책이 생길 때 재검토한다.

### SubscriptionEligibility

| 항목 | 내용 |
| --- | --- |
| 해결하는 문제 | SKU가 구독 가능한지 판단하는 의미 표현 |
| 보장하는 불변 조건 | 구독 생성에는 구독 가능한 SKU만 사용 |
| 값 객체 이점 | 단순 Boolean보다 정책 확장에 유리할 수 있음 |
| 현재 단계의 복잡성 | SKU별 구독 가능 여부 저장 방식과 상태 모델이 미정 |
| 최종 판단 상태 | Deferred |

구독 가능 여부는 우선 SKU가 제공하는 능력 또는 속성으로 둔다. 단순 Boolean, enum, 값 객체와 DB 표현은 DATA 또는 구현 설계에서 확정한다.

## 다음 주문 예정일 계산 책임

Approved Input은 다음과 같다.

```text
다음 주문 예정일 = 구독 생성일 + 배송 주기
```

Accepted Domain Design은 다음과 같다.

- 애플리케이션 경계는 현재 시각 또는 기준 날짜를 Asia/Seoul 기준 날짜로 공급한다.
- 구독 도메인 경계는 생성일과 DeliveryCycle로 다음 주문 예정일을 계산한다.
- 계산 결과는 생성일보다 이후여야 한다.
- 휴일, 주말과 영업일 보정은 하지 않는다.
- 다음 주문 예정일은 표시용 예정 정보이며 정기 주문 자동 생성 트리거가 아니다.

`Clock`, 프레임워크 컴포넌트, 실제 클래스 구조와 테스트 더블 전략은 구현 단계에서 결정한다.

## 소유권과 조회 책임

Subscription은 자신의 소유자를 식별할 수 있어야 한다.

첫 번째 MVP의 도메인 의미는 다음과 같다.

- 자신의 구독인지 비교할 수 있다.
- 다른 회원의 구독은 자신의 목록에 포함되지 않는다.
- 다른 회원의 구독 상세 접근은 조회할 수 없는 구독으로 처리한다.

인증 사용자를 찾는 방식은 도메인 책임이 아니다. 목록 필터링, 조회 쿼리, HTTP 접근 차단과 오류 응답 표현은 애플리케이션 또는 API 설계 책임이다.

## 구독 생성 흐름

### 정상 흐름

```text
인증 회원 식별
→ SKU 조회
→ SKU 구독 가능 여부 확인
→ 수량 생성·검증
→ 배송 주기 생성·검증
→ Asia/Seoul 기준 구독 생성일 공급
→ 다음 주문 예정일 계산
→ 일관된 Subscription 생성
```

### 실패 흐름과 책임

| 실패 | 책임 구분 | 설명 |
| --- | --- | --- |
| 회원을 식별할 수 없음 | 애플리케이션 조정, API 표현 | 인증 실패 처리와 화면 이동 또는 HTTP 응답은 API-001 책임 |
| SKU가 존재하지 않음 | 애플리케이션 조정, API 표현 | 저장소 조회 실패이며 HTTP 표현은 API-001 책임 |
| SKU가 구독 불가능함 | 도메인 불변 조건, API 표현 | 구독 생성에는 구독 가능한 SKU만 허용 |
| 수량 범위 위반 | 도메인 불변 조건 | SubscriptionQuantity가 1~10 범위를 보장 |
| 배송 주기가 허용 범위 밖임 | 도메인 불변 조건 | DeliveryCycle이 2주·4주·8주만 허용 |
| 필수 값 누락 | 애플리케이션 조정, 도메인 불변 조건 | 입력 누락은 API 경계에서 먼저 다루고 도메인 생성도 완전한 값만 허용 |
| 자신의 구독이 아닌 대상을 조회 | 제품 규칙, 애플리케이션 조정, API 표현 | Subscription 소유자 의미를 사용하되 조회 차단 표현은 API-001 책임 |

## Accepted Domain Design

- 첫 번째 MVP에서 회원과 고객은 관점에 따라 구분한다.
- Product는 상품 공통 정보, SKU는 실제 구독 선택 단위로 본다.
- SKU는 구독 가능 여부를 제공한다.
- Subscription은 소유자, SKU 하나, 수량, 배송 주기, 생성일, 다음 주문 예정일을 가진다.
- SubscriptionQuantity와 DeliveryCycle은 값 객체로 다루는 설계를 채택한다.
- 다음 주문 예정일 계산은 구독 도메인 내부 책임으로 둔다.
- Subscription은 소유자를 식별할 수 있어야 한다.
- 첫 번째 MVP에서는 구독 상태 모델을 만들지 않는다.

## Proposed Technical Design

- Product와 SKU를 같은 Aggregate로 둘지 별도 Aggregate로 둘지는 후속 구현 설계에서 결정한다.
- Subscription이 SKU 전체를 참조할지 SKU 식별자와 구독 가능 스냅샷 의미만 가질지는 후속 구현 설계에서 결정한다.
- 날짜 공급을 위한 `Clock` 또는 동등한 추상화는 구현 단계에서 검토한다.
- 구독 가능 여부 표현은 Boolean, enum, 값 객체 중 하나로 구현 단계에서 결정한다.

- 비로그인 접근의 HTTP 상태와 오류 응답
- 존재하지 않는 SKU와 구독 불가능 SKU의 API 표현
- 다른 회원 구독 접근을 조회할 수 없는 구독으로 처리하는 API 표현

- 구독 생성 요청·응답 구조
- 내 구독 목록·상세 응답 구조

- Subscription의 소유자와 SKU 참조 방식

## UX 설계가 알아야 할 제약

- 사용자는 최초 주문 예정일을 선택하지 않는다.
- 선택 가능한 배송 주기는 2주, 4주, 8주다.
- 다음 주문 예정일만 보여주며 배송 예정일은 보여주지 않는다.
- 구독 수량은 1~10이다.
- 구독 불가능한 SKU는 구독 대상으로 사용할 수 없음을 구분해야 한다.
- 첫 번째 MVP에는 구독 상태, 변경, 건너뛰기, 일시정지, 재개, 해지 액션이 없다.
