# ARCH-001 첫 수직 MVP 아키텍처 결정

- 작업 ID: `ARCH-001`
- 역할: Tech Lead
- 결정 상태: Proposed Architecture Decision
- 기준 입력: `PS-002`, `DOMAIN-001`, `UX-001`, `PS-003`, `UX-002`
- 기준 문서:
  - `docs/product/PS-002-first-mvp-requirements.md`

사용자 승인 전까지 이 문서는 `Approved`로 표시하지 않는다.

## 작업 목적

첫 번째 수직 MVP의 상품 탐색, 구독 생성, 내 구독 조회 흐름을 구현하기 전에 시스템 경계와 책임 분리를 정리한다.

이번 결정은 후속 DATA-001, API-001, Backend, Frontend, QA가 제품·도메인·UX 규칙을 추측하지 않도록 인증·인가 경계, 로그인 복귀 경계, 날짜 책임, 오류 처리 경계와 요구사항 추적성을 제공한다.

## 승인된 입력

- 비회원과 로그인 회원은 상품 목록, 상품 상세, SKU와 구독 가능 여부를 확인할 수 있다.
- 구독 생성, 내 구독 목록, 내 구독 상세는 로그인한 회원만 사용할 수 있다.
- 비회원이 보호 기능에 접근하면 로그인 화면으로 이동한다.
- 로그인 성공 후 안전한 서비스 내부 GET 경로가 있으면 해당 경로로 복귀한다.
- 유효한 내부 복귀 경로가 없으면 상품 목록으로 이동한다.
- 외부 URL과 검증되지 않은 임의 경로로 이동하지 않는다.
- 첫 번째 MVP에서는 입력 폼 상태를 자동 복원하지 않는다.
- 이전 POST 요청을 자동으로 다시 실행하지 않는다.
- 구독 하나는 SKU 하나만 대상으로 한다.
- 수량은 1~10이다.
- 배송 주기는 2주, 4주, 8주다.
- 구독 생성 성공 후 생성된 구독 상세 화면으로 이동한다.
- 구독 상세 화면에서 생성 성공 안내, 구독 ID와 다음 주문 예정일을 확인할 수 있어야 한다.
- 별도의 영구적인 구독 생성 성공 페이지는 만들지 않는다.
- 구독 생성 전에는 정확한 다음 주문 예정일 날짜를 표시하지 않는다.
- 구독 생성 성공 후 서버가 계산한 다음 주문 예정일을 표시한다.
- 내 구독 목록과 상세에서도 서버가 확정한 다음 주문 예정일을 표시한다.
- 날짜 계산과 표시는 `Asia/Seoul` 기준 날짜 단위로 다룬다.
- 휴일, 주말, 영업일 보정은 적용하지 않는다.
- 화면 날짜 표시는 `YYYY. M. D.`를 사용한다.
- 다음 배송 예정일은 표시하지 않는다.
- 첫 번째 MVP에는 구독 상태, 결제, 재고, 배송, 구독 변경·해지·일시정지 기능이 없다.

- 상품 상세와 SKU 조회
- SKU 구독 가능 여부 확인
- 구독 생성
- 구독 생성 성공 후 생성된 구독 상세 확인
- 내 구독 목록 조회
- 내 구독 상세 조회
- 비회원 보호 기능 접근 시 로그인 화면 이동
- 로그인 후 안전한 내부 GET 복귀와 상품 목록 fallback
- 다음 주문 예정일 확인

- 신규 의존성

## 아키텍처 원칙

- 첫 번째 수직 MVP는 상품 탐색, 구독 생성, 내 구독 조회를 관통하는 최소 구조로 설계한다.
- 결제, 재고, 배송, 구독 상태 모델은 현재 아키텍처에 포함하지 않는다.
- 기술 도입보다 요구사항 추적성과 검증 가능성을 우선한다.
- API, 데이터, 프론트엔드, 인증 구현 상세는 후속 작업에서 확정한다.
- 도메인 규칙은 백엔드 도메인/애플리케이션 경계에서 일관되게 검증한다.
- 화면은 서버가 확정한 결과를 사용자에게 표시하고, 다음 주문 예정일을 사전 계산하지 않는다.
- 인증·인가 실패는 사용자 흐름과 API 계약에서 구분하되, HTTP 상태와 오류 JSON은 API-001에서 정한다.

## 도메인 책임

- Product는 사용자에게 보여지는 공통 상품 정보와 SKU 목록 또는 SKU별 표시 정보를 제공하는 의미를 가진다.
- SKU는 용량 또는 구성, 표시 가격, 구독 가능 여부를 가진 실제 구독 선택 단위다.
- Subscription은 회원, SKU 하나, 수량, 배송 주기, 구독 생성일, 다음 주문 예정일의 의미를 가진다.
- 구독 하나는 SKU 하나만 대상으로 한다.
- SKU는 구독 가능해야 한다.
- 수량 범위는 1~10이다.
- 배송 주기는 2주, 4주, 8주 중 하나다.
- 다음 주문 예정일은 생성일과 배송 주기에 따라 계산된다.
- 날짜 계산은 `Asia/Seoul` 기준 날짜 단위로 다룬다.
- 휴일, 주말, 영업일 보정을 적용하지 않는다.
- Subscription은 소유자를 식별할 수 있어야 한다.

## 애플리케이션 책임

- 인증된 회원 식별을 구독 생성과 내 구독 조회 흐름에 공급한다.
- 상품 목록·상세 공개 조회를 조정한다.
- 구독 생성 시 SKU가 존재하는지 조회한다.
- SKU의 구독 가능 여부를 확인한다.
- 수량과 배송 주기를 도메인 규칙에 맞게 전달한다.
- 구독 생성일을 `Asia/Seoul` 기준 날짜 단위로 공급한다.
- 구독 생성 트랜잭션을 조정한다.
- 생성된 구독의 식별자와 서버 확정 다음 주문 예정일을 반환할 수 있어야 한다.
- 내 구독 목록과 상세 조회에서 회원 소유자 조건을 적용한다.
- 존재하지 않는 구독과 다른 회원 소유 구독을 제품 규칙상 “조회할 수 없는 구독”으로 통합해 표현할 수 있게 API 경계에 전달한다.

## 프론트엔드 책임

- 상품 목록과 상세 화면을 표시한다.
- SKU 선택, 수량 입력, 배송 주기 선택 UI를 제공한다.
- SKU 미선택, 수량 누락, 수량 범위 위반, 배송 주기 미선택 등 기본 입력 오류를 사용자에게 피드백한다.
- 비회원이 보호 기능에 접근하면 로그인 화면으로 이동하는 흐름을 반영한다.
- 로그인 후 안전한 내부 GET 경로 복귀 흐름을 반영한다.
- 로그인 후 입력 폼 상태를 자동 복원하지 않는다.
- 이전 POST 요청을 자동으로 다시 실행하지 않는다.
- 구독 생성 전 정확한 다음 주문 예정일 날짜를 계산해 표시하지 않는다.
- 서버가 확정한 구독 생성 결과와 다음 주문 예정일을 표시한다.
- 생성 성공 후 생성된 구독 상세 화면으로 이동한다.
- API 계약에 맞춰 요청·응답을 처리한다.

## 인증·인가 경계

공개 접근:

- 상품 목록
- 상품 상세
- SKU와 구독 가능 여부 확인

로그인 필요:

- 구독 생성
- 내 구독 목록
- 내 구독 상세

소유자 검증 필요:

- 내 구독 목록에는 본인 구독만 포함한다.
- 구독 상세는 본인 구독만 조회 가능하다.
- 다른 회원 소유 구독은 조회할 수 없는 구독으로 처리한다.

이번 문서에서 확정하지 않는 항목:

- Spring Security 구체 설정
- 세션 또는 토큰 방식
- HTTP 401, 403, 404 선택
- 오류 코드
- 오류 응답 JSON
- 실제 라우트 문자열

## 로그인 복귀 경계

- 보호 기능 접근으로 로그인 화면에 이동한 경우 로그인 성공 후 안전한 서비스 내부 GET 경로로 복귀한다.
- 유효한 내부 복귀 경로가 없으면 상품 목록으로 이동한다.
- 외부 URL과 검증되지 않은 임의 경로로 이동하지 않는다.
- 입력 폼 상태를 자동 복원하지 않는다.
- 이전 POST 요청을 자동으로 다시 실행하지 않는다.

ARCH-001에서 정하는 경계:

- 로그인 복귀는 인증 설계의 책임이다.
- 복귀 대상은 안전한 내부 GET 경로로 제한한다.
- Open Redirect 방지는 반드시 후속 인증 설계에서 검증해야 한다.
- 구독 생성처럼 상태를 바꾸는 이전 요청은 로그인 후 자동 재실행하지 않는다.

후속 설계로 넘기는 항목:

- 복귀 경로 저장 방식
- 허용 경로 목록 또는 검증 방식
- 구현 위치
- 구체 URL

## 날짜와 시간 책임

- 다음 주문 예정일은 백엔드 도메인 또는 애플리케이션 경계에서 계산한다.
- 프론트엔드는 생성 전 정확한 다음 주문 예정일 날짜를 계산해 표시하지 않는다.
- 프론트엔드는 서버가 확정한 다음 주문 예정일을 표시한다.
- 기준은 `Asia/Seoul` 날짜 단위다.
- 휴일, 주말, 영업일 보정은 적용하지 않는다.
- 화면 표시는 `YYYY. M. D.` 형식을 따른다.
- API 날짜 표현은 API-001에서 확정한다.

후속 설계로 넘기는 항목:

- 날짜 값 객체 추가 여부
- DB 저장 타입
- API 날짜 문자열 형식
- 기준 날짜 공급을 위한 `Clock` 또는 동등한 추상화 사용 여부

## QA 검증으로 넘길 기준

- 비회원과 로그인 회원 모두 상품 목록을 조회할 수 있다.
- 비회원과 로그인 회원 모두 상품 상세, SKU와 구독 가능 여부를 조회할 수 있다.
- 비회원이 구독 생성, 내 구독 목록, 내 구독 상세에 접근하면 로그인 화면으로 이동한다.
- 로그인 성공 후 안전한 내부 GET 복귀 경로가 있으면 해당 경로로 복귀한다.
- 유효한 내부 복귀 경로가 없으면 상품 목록으로 이동한다.
- 외부 URL과 검증되지 않은 임의 경로로 복귀하지 않는다.
- 로그인 후 입력 폼 상태가 자동 복원되지 않는다.
- 이전 POST 요청이 자동 재실행되지 않는다.
- 로그인 회원은 구독 가능한 SKU 하나, 수량 1~10, 배송 주기 2주·4주·8주로 구독을 만들 수 있다.
- 생성 성공 후 생성된 구독 상세로 이동한다.
- 생성된 구독 상세에는 생성 성공 안내, 구독 ID와 다음 주문 예정일이 표시된다.
- 생성 전 입력 검토 단계에는 정확한 다음 주문 예정일 날짜가 표시되지 않는다.
- 내 구독 목록과 상세에는 서버가 확정한 다음 주문 예정일이 표시된다.
- 다른 회원 소유 구독은 조회할 수 없는 구독으로 처리하고 소유자·존재 여부를 노출하지 않는다.
- 다음 배송 예정일, 구독 상태, 결제, 재고, 배송, 변경 액션은 표시되지 않는다.

## 위험과 제한

- 로그인 복귀 정책은 제품 결정으로 승인됐지만 저장 방식과 검증 방식은 아직 기술 설계가 필요하다.
- Open Redirect 방지를 후속 인증 설계에서 놓치면 안전한 내부 GET 복귀 원칙이 깨질 수 있다.
- API 계약이 확정되기 전까지 프론트엔드는 사용자 흐름과 표시 정책만 구현 기준으로 삼아야 한다.
- DB 저장 구조가 확정되기 전까지 Backend는 영속성 구현을 시작할 수 없다.
- 클라이언트가 다음 주문 예정일을 사전 계산하면 서버 확정 값과 불일치할 수 있으므로 FE와 QA에서 금지 기준을 확인해야 한다.
- 다른 회원 소유 구독과 존재하지 않는 구독을 같은 사용자 표현으로 처리해야 하므로 API-001에서 정보 노출 방지 기준을 명확히 해야 한다.

## 후속 작업 순서

1. DATA-001: Product, SKU, Member, Subscription 저장 구조와 조회 인덱스 후보 정의
2. API-001: 공개 상품 조회, 구독 생성, 내 구독 조회, 인증·인가 실패와 오류 계약 정의
3. 인증 설계: 로그인 복귀 경로 저장, 내부 GET 검증, Open Redirect 방지 결정
4. Backend: 도메인·애플리케이션·API 구현과 테스트
5. Frontend: UX-002와 API-001에 맞춘 화면·상태·라우팅 구현
6. QA: 요구사항, ARCH-001, API-001, UX-002 기준 검증 계획과 테스트 작성
