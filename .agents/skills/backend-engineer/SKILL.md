# 백엔드 엔지니어 Skill

## 1. Skill 이름

`backend-engineer`

## 2. Skill 설명

제품 정책, API 계약(API Contract), 트랜잭션 명확성, 테스트 커버리지(Test Coverage)를 지키며 승인된 백엔드 변경을 구현한다.

## 3. 사용하는 상황

- 승인된 요구사항에 백엔드 구현이 필요하다.
- API 동작, 도메인 규칙, 영속성, 보안이 변경되어야 한다.
- 백엔드 테스트나 버그 수정이 필요하다.
- 프론트엔드, QA, SRE가 백엔드 인수인계를 필요로 한다.

## 4. 사용하지 않는 상황

- 제품 요구사항 또는 API 계약이 승인되지 않았다.
- 프론트엔드 구현 작업이다.
- 인프라 구현 작업이다.
- 측정 근거 없는 성능 최적화 요청이다.

## 5. 작업 전 확인할 자료

1. `AGENTS.md`
2. `backend/AGENTS.md`
3. `docs/roles/backend-engineer.md`
4. 승인된 제품 요구사항과 인수 조건
5. `docs/domain/glossary.md`와 `docs/domain/rules.md`
6. `docs/adr/**`의 승인된 ADR
7. `docs/api/**`의 승인된 API 계약
8. 기존 백엔드 코드와 테스트

## 6. 단계별 작업 절차

1. 작업 ID, 범위, 승인된 입력을 확인한다.
2. 구현을 막는 Product Decision이 남아 있으면 중단한다.
3. Technical Decision이 승인됐는지 확인하거나 작은 결정이면 문서화한다.
4. 수정 전에 기존 백엔드 관례를 확인한다.
5. 인수 조건을 만족하는 가장 작은 변경을 설계한다.
6. 트랜잭션 경계와 영속성 영향을 명시한다.
7. 중복 요청, 구독, 결제 동작이 있으면 멱등성과 동시성을 검토한다.
8. 허용 경로 안에서만 코드를 구현한다.
9. 집중 테스트를 추가하거나 갱신한다.
10. API 문서, DB 메모, ADR, 인수인계를 영향 범위에 맞게 갱신한다.
11. 관련 검사를 실행하고 실행할 수 없는 검사는 이유를 보고한다.

## 7. 허용 경로

- `backend/**`
- `docs/api/**`
- `docs/adr/**`
- `docs/handoffs/**`
- 승인된 범위의 `docs/domain/**`

## 8. 금지 경로

- `frontend/**`
- `infra/**`
- 승인되지 않은 비즈니스 정책 변경
- 승인되지 않은 프로덕션 의존성
- 측정되지 않은 성능 최적화
- 범위를 벗어난 광범위한 리팩터링

## 9. 필수 검증

- 백엔드 프로젝트가 있으면 관련 백엔드 테스트를 실행한다.
- 사용 가능하고 관련 있으면 빌드(Build) 또는 Gradle 검사를 실행한다.
- API 변경 시 문서와 동작이 일치하는지 확인한다.
- 영속성 변경 시 DB 변경이 문서화됐는지 확인한다.
- 관련될 경우 인수인계에 트랜잭션, 멱등성, 위험 메모가 포함됐는지 확인한다.

## 10. 필수 산출물

- 백엔드 코드
- 단위 테스트 또는 통합 테스트
- 필요 시 API 문서 변경
- 필요 시 DB 변경 메모
- 장기 기술 결정이 있으면 ADR
- 필요 시 프론트엔드, QA, SRE 인수인계

## 11. 인수인계 대상

- 프론트엔드 엔지니어: API 사용과 응답 동작
- QA 엔지니어: 검증 시나리오와 알려진 위험
- 플랫폼/SRE: 성능 또는 운영 메모
- 기획자: 해결되지 않은 정책 결정

## 12. 완료 보고 형식

다음을 보고한다.

- 변경한 백엔드 파일
- 구현한 동작
- 트랜잭션과 영속성 메모
- API 또는 DB 영향
- 실행한 테스트와 검사
- 생성한 인수인계
- 위험과 열린 결정

## 13. 중단하고 사용자 결정을 요청해야 하는 조건

- 비즈니스 정책이 승인되지 않았다.
- API 또는 DB 설계에 새 결정이 필요하다.
- 새 의존성이 필요하다.
- 성능 변경에 측정 근거가 없다.
- 요청된 수정에 프론트엔드 또는 인프라 코드 변경이 필요하다.
