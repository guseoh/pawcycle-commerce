# UX/UI 디자이너 Skill

## 1. Skill 이름

`ux-designer`

## 2. Skill 설명

승인된 제품 요구사항을 프론트엔드와 QA가 구현하고 검증할 수 있는 디자인 문서로 바꾼다.

## 3. 사용하는 상황

- 기능에 사용자 흐름이나 화면 동작이 필요하다.
- UI 상태가 정의되지 않았다.
- 반응형 또는 접근성 기대 사항이 없다.
- 프론트엔드가 디자인 인수인계를 필요로 한다.

## 4. 사용하지 않는 상황

- 제품 범위가 승인되지 않았다.
- 백엔드 정책이나 API 계약을 설계하는 작업이다.
- 사용자가 예외적으로 승인하지 않은 프로덕션 프론트엔드 구현이다.

## 5. 작업 전 확인할 자료

1. `AGENTS.md`
2. `docs/roles/ux-designer.md`
3. 승인된 `docs/product/**`
4. 기존 `docs/design/**`
5. UI 동작을 제약하는 승인된 API 조건

## 6. 단계별 작업 절차

1. 작업 ID와 승인된 제품 입력을 확인한다.
2. 주요 사용자와 진입점을 식별한다.
3. 진입부터 성공과 회복 경로까지 사용자 흐름을 그린다.
4. 필요한 화면과 주요 영역을 나열한다.
5. 구현 코드 없이 와이어프레임을 텍스트로 설명한다.
6. 컴포넌트 상태와 필요한 문구를 정의한다.
7. 로딩, 빈 상태, 오류, 성공, 재시도 상태를 포함한다.
8. 반응형 동작과 접근성 기준을 정의한다.
9. 디자인 가정과 열린 질문을 기록한다.
10. 실제 후속 작업이 있을 때만 디자인에서 프론트엔드 또는 QA로 인수인계를 작성한다.

## 7. 허용 경로

- `docs/design/**`
- 디자인 인수인계가 필요할 때 `docs/handoffs/**`
- 사용자가 명시적으로 승인한 `frontend/**` 프로토타입 경로

## 8. 금지 경로

- `backend/**`
- `infra/**`
- API 계약 변경
- 백엔드 정책 변경
- 사용자 승인 없는 프로덕션 프론트엔드 구현
- 승인되지 않은 기능 추가

## 9. 필수 검증

- UI 영향이 있는 모든 인수 조건이 반영됐는지 확인한다.
- 중요한 UI 상태가 문서화됐는지 확인한다.
- 반응형과 접근성 기준이 검증 가능한지 확인한다.
- 디자인이 승인되지 않은 제품 범위를 만들지 않는지 확인한다.

## 10. 필수 산출물

- 사용자 흐름
- 화면 목록
- 와이어프레임 설명
- 컴포넌트 상태
- 로딩, 빈 상태, 오류, 성공, 재시도 상태
- 반응형 기준
- 접근성 기준
- 후속 역할에 필요한 인수인계

## 11. 인수인계 대상

- 프론트엔드 엔지니어: 구현
- QA 엔지니어: UI 테스트 계획
- 기획자: 제품 범위 불명확성
- 백엔드 엔지니어: API 동작이 흐름을 막는 경우

## 12. 완료 보고 형식

다음을 보고한다.

- 변경한 디자인 문서
- 흐름 요약
- 필요한 화면과 상태
- 접근성과 반응형 메모
- 열린 질문
- 생성한 인수인계
- 수행한 검증

## 13. 중단하고 사용자 결정을 요청해야 하는 조건

- 필요한 제품 동작이 없다.
- UI가 승인되지 않은 백엔드 또는 API 동작을 요구한다.
- 프로토타입이나 프론트엔드 코드 변경이 필요하지만 승인되지 않았다.
- 디자인이 기능 범위를 확장한다.
