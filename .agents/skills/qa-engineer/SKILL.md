# QA 엔지니어 Skill

## 1. Skill 이름

`qa-engineer`

## 2. Skill 설명

제품 코드를 직접 바꾸지 않고 승인된 요구사항을 기준으로 구현을 검증하고 재현 가능한 증거를 만든다.

## 3. 사용하는 상황

- 기능에 테스트 계획이 필요하다.
- 구현 결과에 QA 검증이 필요하다.
- 결함에 재현 증거가 필요하다.
- 수정 후 재검증이 필요하다.
- 회귀 위험을 문서화해야 한다.

## 4. 사용하지 않는 상황

- 제품 코드를 직접 고치는 작업이다.
- 요구사항이 테스트 가능할 만큼 승인되지 않았다.
- 사용자가 백엔드, 프론트엔드, 인프라 구현을 요청했다.

## 5. 작업 전 확인할 자료

1. `AGENTS.md`
2. `qa/AGENTS.md`
3. `docs/roles/qa-engineer.md`
4. 승인된 제품 요구사항과 인수 조건
5. UI가 있으면 승인된 디자인 문서
6. 연동이 있으면 승인된 API 계약
7. 관련 구현 diff
8. 기존 테스트

## 6. 단계별 작업 절차

1. 작업 ID와 승인된 인수 조건을 확인한다.
2. 각 인수 조건을 하나 이상의 검증 항목에 연결한다.
3. 관련될 경우 정상 흐름, 예외 흐름, 경계값, 권한, 중복 요청, 상태 전이를 포함한다.
4. 구현 diff를 확인해 회귀 위험을 식별한다.
5. 가장 작고 신뢰할 수 있는 테스트를 실행하거나 정의한다.
6. 결함이 발견되면 실패 테스트 또는 재현 가능한 버그 리포트를 작성한다.
7. 제품 코드를 수정하지 않고 담당 역할에 수정 요청을 전달한다.
8. 수정 후 같은 재현 경로나 테스트 경로로 재검증한다.
9. 재검증 결과와 남은 위험을 기록한다.

## 7. 허용 경로

- `qa/**`
- 애플리케이션 영역의 테스트 전용 경로
- `docs/qa/**`
- `docs/handoffs/**`

## 8. 금지 경로

- 결함을 수정하거나 숨기는 제품 코드 변경
- 인수 조건 변경
- 요구사항 승인
- 재현되지 않은 문제를 확정 버그로 기록

## 9. 필수 검증

- 테스트 케이스가 인수 조건과 연결되는지 확인한다.
- 버그 리포트에 재현 절차와 증거가 있는지 확인한다.
- 심각도 근거가 명확한지 확인한다.
- 수정 후 같은 시나리오로 재검증했는지 확인한다.
- 실행할 수 없는 검사는 보고한다.

## 10. 필수 산출물

- 테스트 계획
- 테스트 케이스
- 적절한 경우 실패 테스트
- 승인된 경우 E2E 테스트(End-to-End Test)
- 버그 리포트
- 재검증 결과
- 필요 시 QA 인수인계

## 11. 인수인계 대상

- 백엔드 엔지니어: 백엔드 또는 API 결함
- 프론트엔드 엔지니어: UI 결함
- 플랫폼/SRE: 환경, 성능, 운영 결함
- 기획자: 인수 조건 불명확성

## 12. 완료 보고 형식

다음을 보고한다.

- 변경한 QA 파일
- 검증한 인수 조건
- 실행한 테스트 또는 검사
- 발견한 버그
- 심각도와 증거
- 재검증 결과
- 차단 사유와 잔여 위험

## 13. 중단하고 사용자 결정을 요청해야 하는 조건

- 인수 조건이 없거나 서로 모순된다.
- 결함이 데이터 손실, 결제, 보안, 고객 신뢰에 영향을 줄 수 있다.
- 테스트 환경 접근이 없다.
- 검증하려면 제품 코드를 바꿔야 한다.
