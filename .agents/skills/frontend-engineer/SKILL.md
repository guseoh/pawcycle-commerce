# 프론트엔드 엔지니어 Skill

## 1. Skill 이름

`frontend-engineer`

## 2. Skill 설명

제품, 디자인, API 입력을 기준으로 승인된 프론트엔드 동작을 구현하고 서버 비즈니스 규칙은 백엔드에 남긴다.

## 3. 사용하는 상황

- 승인된 디자인에 프론트엔드 구현이 필요하다.
- Next.js 페이지, React 컴포넌트, TypeScript 타입을 변경해야 한다.
- 승인된 API 계약에 따른 연동이 필요하다.
- 프론트엔드 테스트나 버그 수정이 필요하다.

## 4. 사용하지 않는 상황

- 디자인 또는 API 계약이 승인되지 않았다.
- 백엔드 구현 작업이다.
- 인프라 구현 작업이다.
- UI에 서버 정책을 중복 구현해야만 하는 변경이다.

## 5. 작업 전 확인할 자료

1. `AGENTS.md`
2. `frontend/AGENTS.md`
3. `docs/roles/frontend-engineer.md`
4. 승인된 제품 요구사항
5. `docs/design/**`의 승인된 디자인 문서
6. `docs/api/**`의 승인된 API 계약
7. `docs/handoffs/**`의 관련 백엔드 인수인계
8. 기존 프론트엔드 코드와 테스트

## 6. 단계별 작업 절차

1. 작업 ID, 범위, 승인된 디자인, API 계약을 확인한다.
2. 누락된 API 동작이나 디자인 상태가 있으면 코딩 전에 식별한다.
3. 수정 전에 기존 프론트엔드 관례를 확인한다.
4. 인수 조건을 만족하는 가장 작은 UI 변경을 구현한다.
5. 서버 상태(Server State), 로컬 UI 상태(Local UI State), 폼 상태(Form State)를 개념적으로 분리한다.
6. 서버가 제공한 비즈니스 결과를 표시하고 프론트엔드에서 재계산하지 않는다.
7. 관련될 경우 로딩, 빈 상태, 오류, 성공, 재시도 상태를 구현한다.
8. 컴포넌트 또는 흐름에 맞는 접근성을 추가한다.
9. 집중 테스트를 추가하거나 갱신한다.
10. 후속 작업이 필요하면 QA 또는 백엔드 인수인계를 작성한다.
11. 관련 검사를 실행하고 실행할 수 없는 검사는 이유를 보고한다.

## 7. 허용 경로

- `frontend/**`
- `docs/handoffs/**`
- 승인된 프론트엔드 관련 문서

## 8. 금지 경로

- `backend/**`
- `infra/**`
- 승인 없는 API 계약 변경
- 가격, 할인, 재고, 결제, 구독 정책 중복 구현
- 승인 없는 디자인 변경
- 승인 없는 의존성 추가

## 9. 필수 검증

- 프론트엔드 프로젝트가 있으면 타입 검사(Type Check)를 실행한다.
- 사용 가능하고 관련 있으면 린트(Lint), 테스트, 빌드를 실행한다.
- UI 상태가 승인된 디자인과 일치하는지 확인한다.
- API 사용이 승인된 계약과 일치하는지 확인한다.
- 서버 비즈니스 규칙을 프론트엔드 로직에 중복하지 않았는지 확인한다.

## 10. 필수 산출물

- 승인된 경우 Next.js 페이지 또는 라우트(Route)
- React 컴포넌트
- TypeScript 타입
- API 연동
- 컴포넌트 또는 흐름 테스트
- 필요 시 프론트엔드에서 QA로 전달하는 인수인계

## 11. 인수인계 대상

- QA 엔지니어: 검증 시나리오
- 백엔드 엔지니어: API 동작 누락 또는 불일치
- UX/UI 디자이너: 디자인 상태 누락
- 기획자: 제품 문구 또는 정책 불명확성

## 12. 완료 보고 형식

다음을 보고한다.

- 변경한 프론트엔드 파일
- 구현한 UI 동작
- API 사용 방식
- 처리한 상태
- 접근성 메모
- 실행한 테스트와 검사
- 생성한 인수인계
- 위험과 열린 결정

## 13. 중단하고 사용자 결정을 요청해야 하는 조건

- API 동작을 추측해야 한다.
- 승인된 디자인에 필요한 상태가 없다.
- 비즈니스 정책을 프론트엔드 로직에 구현해야 한다.
- 새 의존성 또는 상태 관리 라이브러리가 필요하다.
- 요청된 수정에 백엔드 또는 인프라 변경이 필요하다.

## 14. 공통 운영 기준

- 공통 Git, commit·push, 작업 보고서, 인수인계 규칙은 루트 `AGENTS.md`를 따른다.
- 프론트엔드 역할 브랜치는 `feat/fe`다.
- 하나의 역할 브랜치에는 하나의 활성 작업만 둔다.
