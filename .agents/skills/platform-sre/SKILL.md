# 플랫폼/SRE Skill

## 1. Skill 이름

`platform-sre`

## 2. Skill 설명

운영과 성능 작업을 요구사항, 측정 근거, 사용자 승인에 기반해 수행한다.

## 3. 사용하는 상황

- 로컬 개발 환경 또는 인프라 작업이 승인됐다.
- CI/CD 또는 배포 설정이 승인됐다.
- 성능 측정 또는 부하 테스트가 요청됐다.
- 관측성, 알림, 런북 작업이 필요하다.
- 운영 인수인계 또는 개선 요청이 필요하다.

## 4. 사용하지 않는 상황

- 애플리케이션 기능 구현 작업이다.
- 기준 성능 없이 성능 최적화가 요청됐다.
- 필요한 비밀 값이나 운영 접근 권한이 없다.
- 백엔드 또는 프론트엔드 제품 코드를 직접 수정해야 한다.

## 5. 작업 전 확인할 자료

1. `AGENTS.md`
2. `infra/AGENTS.md`
3. `docs/roles/platform-sre.md`
4. 승인된 운영 요구사항
5. 기존 측정 결과 또는 장애 증거
6. 관련 백엔드, 프론트엔드, QA 인수인계
7. 기존 인프라 파일

## 6. 단계별 작업 절차

1. 작업 ID, 운영 목적, 승인 범위를 확인한다.
2. 기준 성능(Baseline), 목표(Target), 환경(Environment), 측정 방법을 식별한다.
3. 필요한 비밀 값은 문서화하되 저장소에 저장하지 않는지 확인한다.
4. 성능 작업은 최적화 제안 전에 먼저 측정한다.
5. 증거를 수집하고 병목 가설(Bottleneck Hypothesis)을 작성한다.
6. 승인된 인프라 또는 운영 변경만 구현한다.
7. 애플리케이션 코드 변경이 필요하면 백엔드 또는 프론트엔드에 인수인계로 요청한다.
8. 운영 변경에는 롤백 메모를 포함한다.
9. 변경 후 동일 조건으로 재측정한다.
10. 결과, 제한, 다음 작업을 기록한다.

## 7. 허용 경로

- `infra/**`
- `.github/workflows/**`
- `docs/performance/**`
- `docs/runbook/**`
- `docs/handoffs/**`
- 승인된 운영 설정 파일

## 8. 금지 경로

- 백엔드 제품 코드
- 프론트엔드 제품 코드
- 비밀 값
- 측정 없는 성능 최적화
- 롤백 메모 없는 배포 변경
- 결과를 유리하게 만드는 부하 조건 변경

## 9. 필수 검증

- 기준 측정과 변경 후 측정이 비교 가능한 조건인지 확인한다.
- 저장소에 비밀 값이 없는지 확인한다.
- 운영 변경에 롤백 메모가 있는지 확인한다.
- 도구가 있으면 문법 또는 설정 검사를 실행한다.
- 실행할 수 없는 검사는 보고한다.

## 10. 필수 산출물

- 승인된 경우 Docker, CI/CD, 배포, 관측성 설정
- 성능 실험 문서
- 승인된 경우 k6 시나리오
- 런북
- 대시보드 또는 알림 메모
- 운영 인수인계 또는 애플리케이션 개선 요청

## 11. 인수인계 대상

- 백엔드 엔지니어: 서버 병목 또는 API 신뢰성 문제
- 프론트엔드 엔지니어: 클라이언트 성능 문제
- QA 엔지니어: 재현 가능한 환경 문제
- 사용자/Tech Lead: 비용, 의존성, 배포, 비밀 값 결정

## 12. 완료 보고 형식

다음을 보고한다.

- 변경한 운영 파일
- 기준 성능과 목표
- 측정 방법
- 적용한 변경
- 검증 결과
- 롤백 메모
- 생성한 인수인계
- 위험과 필요한 결정

## 13. 중단하고 사용자 결정을 요청해야 하는 조건

- 새로운 유료 서비스, 의존성, 인프라 컴포넌트가 필요하다.
- 비밀 값 또는 운영 접근 권한이 필요하다.
- 롤백 방법을 정의할 수 없다.
- 최적화 요청에 측정 근거가 없다.
- 애플리케이션 제품 코드 변경이 필요하다.
